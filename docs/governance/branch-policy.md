# MyBlog 分支治理策略

> 状态：当前有效
> 适用范围：Git 分支、提交和远程协作
> 最后校准：2026-07-10
> 对应代码：`.github/workflows/ci.yml`、`AGENTS.md`
> 权威程度：治理规则

## 1. 分支职责

| 分支 | 职责 |
| --- | --- |
| `main` | V2 唯一日常主线和稳定基线 |
| `feature/*` | 单一功能任务 |
| `fix/*` | 单一缺陷修复 |
| `refactor/*` | 单一重构任务 |
| `docs/*` | 单一文档任务 |
| `release/*` | 发布期临时冻结和修复；发布完成后删除或打标签 |
| `archive/*` | 只读历史分支，不参与开发 |

个人项目当前不维护长期集成分支。短期任务从 `main` 创建，验证后合回 `main`。只有明确出现多任务并行集成需求时，才评估新增集成分支。

## 2. 命名

任务分支使用 `<type>/<scope>-<short-name>`：

- `feature/blog-password-unlock`
- `fix/backend-refresh-race`
- `refactor/admin-http-client`
- `docs/release-guide`

分支名必须表达单一目的，不使用工具名称、阶段状态或个人名称。

## 3. 提交

1. 每个提交只完成一个可独立验证、独立回滚的目的。
2. 提交信息使用中文并说明结果，不使用阶段编号代替内容。
3. 清理、基础设施、业务功能和文档治理不能混在同一提交。
4. 提交前检查 `git diff --stat`、`git status --short` 和任务范围内的验证结果。
5. 批量机械删除或迁移可以单独形成一个提交，但必须与内容改写分开。

## 4. 验证

- 后端变更运行相关 Maven 测试；阶段结束运行 `mvn clean test`。
- blog 变更运行相关测试，并至少执行 `pnpm typecheck` 和 `pnpm build`。
- admin 变更运行相关测试，并至少执行 `pnpm typecheck` 和 `pnpm build`。
- 文档变更检查链接、代码路径、状态头、重复内容和过期引用。

## 5. 主线和远程安全

1. 默认通过短期任务分支进入 `main`；只有获得明确授权时才直接在 `main` 工作。
2. 未经明确授权不得推送远程、删除远程分支、创建标签或发布版本。
3. `main` 不允许强制推送。
4. 其他共享分支需要改写历史时，只允许在明确协调后使用 `--force-with-lease`。
5. 已合入的短期任务分支应删除；历史追溯依赖提交、标签和只读 `archive/*` 分支。
