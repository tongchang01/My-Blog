# V2 业务规则

> 本文档回答："V2 的关键状态机、校验规则和业务不变量是什么？"
> 范围：只描述业务规则；具体接口契约、表结构和代码实现分别见 `api-contract/`、`arch/schema-design.md` 和 V2 工程。

## 1. 全局规则

### BR-001 语言集合固定为三语

- 支持语言固定为 `zh / ja / en`。
- URL 必须带语言前缀。
- 展示型业务内容优先取当前语言字段，缺失时 fallback 到中文。
- 文章正文只维护中文 Markdown，不做三语正文副本。

### BR-002 时间语义固定为 JST

- 所有业务时间按 Asia/Tokyo 解释。
- API 返回的本地时间字符串不带时区后缀，但语义固定为 JST。
- 定时发布、统计日期和清理任务均按 JST 计算。

### BR-003 删除默认是软删除

- 文章、分类、标签、评论、站点配置、附件、友链、用户资料等业务实体默认软删除。
- 软删除必须同时记录删除标记、删除时间和删除人。
- 关联表、token、访问明细、日聚合和邮件日志属于显式例外。

### BR-004 引用完整性由应用层维护

- 业务引用必须在写入前校验目标存在。
- 数据库不使用外键和级联。
- 父实体软删除后，子实体引用保留；展示时按业务规则过滤或 fallback。

## 2. 身份与权限

### BR-101 账号类型只有三态

- ADMIN：站长账号，可执行全部后台写操作。
- DEMO：演示账号，只读后台。
- GUEST：游客身份，用于评论和公开访问，不是可登录后台的普通用户体系。

### BR-102 DEMO 不得写入业务数据

- DEMO 可以读取后台页面和数据。
- DEMO 不能新增、修改、删除、审核、发布、上传或改配置。
- 所有后台写操作必须要求 ADMIN。

### BR-103 登录失败会锁定账号

- 同 IP + 同用户名连续失败达到阈值后，账号进入短期锁定。
- 锁定状态必须能跨进程重启保留。
- 锁定期间拒绝登录，不暴露用户名是否存在的内部判断细节。

### BR-104 token 撤销依赖版本号和 refresh token

- access token 必须携带用户 id、角色、过期时间和 token version。
- refresh token 明文只返回给客户端，服务端只保存 SHA-256 hash。
- 修改密码、全端退出或管理员强制撤销时，用户 token version 自增。
- token version 不一致的 access token 必须失效。

## 3. 文章

### BR-201 文章状态机

| 状态 | 含义 | GUEST 可见 | 可编辑 | 进入方式 |
|------|------|------------|--------|----------|
| DRAFT | 草稿 | 否 | 是 | 新建或 ADMIN 手动切换 |
| PUBLISHED | 已发布 | 是 | 是 | ADMIN 发布或定时任务发布 |
| PRIVATE | 私密 | 否 | 是 | ADMIN 手动切换 |
| PASSWORD | 密码文章 | 列表可见，正文需密码 | 是 | ADMIN 设置访问密码 |
| SCHEDULED | 定时发布 | 未到时间不可见 | 是 | ADMIN 设置发布时间 |

### BR-202 发布状态字段约束

- PUBLISHED、PRIVATE、PASSWORD、SCHEDULED 必须有分类。
- PASSWORD 必须有访问密码哈希。
- 非 PASSWORD 状态不得保留访问密码。
- SCHEDULED 必须有未来或当前的发布时间。
- 非 SCHEDULED 状态不得依赖发布时间决定可见性。

### BR-203 文章 URL 以 id 为主

- 文章公开路径以 id 定位。
- slug 只增强可读性，不作为唯一业务标识。
- slug 可为空、可修改、不强制唯一。
- 分类和标签 slug 必须唯一。

### BR-204 文章可见性

- GUEST 只能看到 PUBLISHED 和已解锁的 PASSWORD 正文。
- PRIVATE、DRAFT、未到发布时间的 SCHEDULED 只允许 ADMIN 在后台查看。
- DEMO 可在后台查看文章数据，但不能改。
- Sitemap 不输出不可公开访问的正文。

### BR-205 评论计数只统计公开评论

- 文章评论数只统计 `target_type=ARTICLE`、目标为当前文章、未删除且审核通过的评论。
- PENDING、HIDDEN 和软删除评论不计入公开评论数。

## 4. 评论与留言板

### BR-301 评论目标用二元定位

- 评论目标由 `target_type + target_id` 决定。
- ARTICLE 评论的目标必须是存在且允许评论的文章。
- GUESTBOOK 留言的目标 id 固定为 0。
- 不允许跨目标回复。

### BR-302 评论最多两层

- 顶层评论没有 parent 和 reply_to。
- 回复顶层评论时，parent 和 reply_to 都指向该顶层评论。
- 回复子评论时，parent 指向同一顶层评论，reply_to 指向实际被回复评论。
- 不允许回复 PENDING、HIDDEN 或已删除评论。

### BR-303 评论审核状态

| 状态 | 前台展示 | 后台展示 | 进入方式 |
|------|----------|----------|----------|
| PASS | 是 | 是 | 未命中黑名单或 ADMIN 审核通过 |
| PENDING | 否 | 是 | 命中关键词黑名单 |
| HIDDEN | 否 | 是 | ADMIN 手动隐藏 |

### BR-304 评论安全基线

- 昵称、邮箱、正文必填；个人站点可选。
- 评论 Markdown 原文必须保存。
- 前台只渲染后端清洗后的安全 HTML。
- 原始 HTML、`javascript:`、`data:`、`vbscript:` 等不可信内容必须被拒绝或清洗。
- 评论提交必须限流。
- 同 IP 在短时间内重复提交相同内容必须拒绝。

### BR-305 PASSWORD 文章评论受密码保护

- PASSWORD 文章的评论列表和评论提交入口都必须先通过文章密码校验。
- 未解锁时不得暴露评论内容。

## 5. 分类与标签

### BR-401 分类是平铺结构

- 分类不支持树形层级。
- 分类按 sort_order 排序。
- 分类名称支持三语副本。

### BR-402 标签是白名单

- 标签由 ADMIN 管理，不允许访客自由创建。
- 标签名称支持三语副本。
- 标签 slug 唯一。

## 6. 站点配置、附件和友链

### BR-501 站点配置只有一行

- 站点配置是宽表单行模型。
- 展示型字段支持三语副本。
- 非展示型字段保持单值。
- 站点配置不得被普通后台操作物理删除。

### BR-502 附件按内容去重

- 上传附件必须计算 SHA-256。
- 相同 SHA-256 的附件视为同一内容。
- 命中软删除附件时优先恢复已有记录。
- 附件必须校验 MIME 和大小。

### BR-503 友链不含申请审核状态

- 友链只有显示和隐藏状态。
- 友链申请走邮件或 GitHub issue，不入当前主表。
- 未来如需在线申请，单独建申请实体，不污染友链主实体。

## 7. 统计与日志

### BR-601 访问统计自研为主

- V2 记录页面访问明细和日聚合。
- 访客标识使用不可逆 hash，不保存可公开展示的个人身份。
- 详细统计第三方服务不进入 V2 schema。

### BR-602 短期数据可物理清理

- refresh token 过期或撤销后可由清理任务物理删除。
- 页面访问明细只保留短期，长期趋势看日聚合。
- 邮件失败日志只保留短期。

### BR-603 邮件只记录失败日志

- 评论回复邮件发送失败后记录日志。
- 成功邮件默认不入库。
- 失败原因必须脱敏，不返回给访客。

## 8. 后台操作体验规则

### BR-701 写操作必须可确认和可反馈

- 删除、隐藏、发布、改密码等危险操作必须二次确认。
- 写操作必须有 loading、成功反馈和失败错误提示。
- 业务失败不得伪装成 HTTP 200 成功。

### BR-702 后台菜单静态定义

- 后台菜单由前端静态路由定义。
- 后端不提供动态菜单接口。
- 权限判断仍以后端鉴权为准。
