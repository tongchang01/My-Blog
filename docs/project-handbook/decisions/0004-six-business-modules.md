# ADR-0004: 划分为六大业务模块

- 状态：accepted（system 模块尚未实现）
- 日期：2026-04
- 决策者：项目负责人

## 背景

V1 表结构按功能聚集（user/article/comment/config/file 等）。V2 模块化时需决定划分粒度。过粗失去模块化意义；过细增加跨模块沟通成本。

## 决定

划分为六大模块：

| 模块 | 职责 | 状态 |
|------|------|------|
| `common` | 跨模块基础设施（响应封装、异常、安全工具） | ✅ 已实现 |
| `infrastructure` | 全局基础设施（MyBatis-Plus 配置、Flyway 配置等） | ✅ 已实现 |
| `identity` | 用户、角色、登录、JWT | ✅ 已实现 |
| `content` | 文章、分类、标签 | ✅ 已实现 |
| `comment` | 评论、审核、举报 | ✅ 已实现 |
| `system` | 系统配置、字典、菜单等后台基础数据 | ⏳ 待实现 |

## 理由

- 按业务边界，非技术边界
- 与 V1 表分组对应，迁移路径清晰
- 评论独立模块：评论场景复杂（审核、回复、软删除），独立后职责清晰
- system 单独：避免散落在其它模块

## 后果

正面：模块边界清晰，跨模块依赖少
负面：跨模块查询（如评论显示用户名）需要明确通信方式（见 `rules/package-layout.md` §跨模块通信）

后续需关注：
- system 模块何时落地
- 跨模块查询逐步收敛到明确接口
