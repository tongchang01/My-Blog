# 博客后台基础工程设计

> 日期：2026-06-20  
> 状态：已确认，待编写实施计划

## 1. 目标

以 Pure Admin Thin 的 i18n 版本为基础，在 `frontend/apps/admin/` 建立可独立运行、验证和部署的博客后台工程。第一阶段只交付后台基础闭环：三语界面、静态路由、登录会话、当前管理员资料、ADMIN/DEMO 权限基线、空仪表盘和工程质量基线。

本阶段不实现文章、分类、标签、附件、站点配置、评论、友链或统计页面，也不修改数据库结构。这些业务模块后续分别设计、实施和提交。

## 4. 模板裁剪

保留：

- 登录页和后台主布局
- 主题与布局偏好
- 多标签页基础能力
- 前端静态路由与路由守卫
- Element Plus、Pinia、Axios、Vue Router 和 vue-i18n
- 403、404 和基础异常页面
- lint、类型检查、样式检查和构建配置

删除：

- Mock 服务及其演示账号
- 演示页面、示例接口和无关素材
- 后端动态路由和动态菜单请求
- 与本阶段无关的大型组件和业务依赖
- 上游部署配置、赞助信息和不适用于本项目的仓库自动化

裁剪分两步进行：先验证原始固定快照可以冻结安装、类型检查和构建，再删除无关内容并重新验证。不得在未建立原始基线前批量裁剪。

## 5. 路由与权限

后台使用前端静态路由，不新增菜单表、角色权限表或菜单接口。`/login` 是唯一匿名页面，其余页面必须有有效会话。

角色仅使用后端现有的 `ADMIN` 与 `DEMO`：

- ADMIN：允许读取和写入授权资源
- DEMO：允许进入后台和读取授权资源，所有写操作保持只读

角色必须通过 `GET /api/auth/me` 获取，不从用户名、token 文本或前端配置推断。路由元数据负责页面访问提示；后续页面的按钮和表单负责只读交互。前端权限仅改善体验，后端 `403 + 10003` 始终是最终权限边界。

第一阶段菜单只包含仪表盘和个人资料入口。后续业务模块按独立规格逐项加入静态菜单。

## 6. 国际化

后台界面支持 `zh`、`ja`、`en`。保留 Pure Admin Thin i18n 版本的中文和英文能力，并补充完整日文资源及 Element Plus 日文 locale。

- 语言选择持久化
- 首次访问根据系统语言映射到 `zh`、`ja` 或 `en`
- 无法映射时使用中文
- 路由标题、菜单、按钮、校验、通知和错误状态都使用资源键
- 前台文章内容的三语字段与后台界面语言相互独立

API 逻辑只判断响应 `code`，不得依赖或直接展示后端中文 `msg`。

## 7. 认证会话

登录流程：

1. 提交 `POST /api/auth/login`。
2. 保存 access token、refresh token、各自过期时间。
3. 调用 `GET /api/auth/me` 获取账号、角色和资料。
4. 成功后进入仪表盘；任一步失败都不得留下半初始化会话。

所有受保护请求统一携带 `Authorization: Bearer <access-token>`。access token 即将过期或受保护请求返回可刷新的认证失败时，客户端调用 `POST /api/auth/refresh`。

刷新必须使用单一并发任务：同一时间只有一个 refresh 请求，其余请求等待该结果。刷新成功后原子替换整对 token，并且每个原请求最多重放一次；刷新请求本身不得进入重试链。刷新失败、refresh token 缺失或收到 `401 + 10002` 时，清理全部会话并跳转登录页。

退出调用 `POST /api/auth/logout`。无论服务端请求成功、超时还是 access token 已失效，本地会话最终都必须清理。token、密码和认证请求体不得输出到控制台或持久化日志。

## 8. 错误处理

HTTP 层将网络错误、HTTP 状态和业务码统一转换为前端错误类型。页面和 Store 不直接解析 Axios 原始异常。

| code / 场景 | 前端行为 |
|---|---|
| `90001` | 显示当前语言的参数错误 |
| `10001` | 登录页显示统一的账号或密码错误 |
| `10002` | 执行受控刷新；失败后返回登录页 |
| `10003` | 保持登录并显示只读账号提示 |
| `90002` | 登录按钮进入冷却状态并提示稍后重试 |
| `99999` | 显示通用服务错误并允许重试 |
| 网络断开、超时 | 显示可重试网络错误 |

并发失败应按错误事件去重，避免同一故障产生多条通知。登录错误不得区分未知账号、GUEST、密码错误或账号锁定。

## 9. 开发与部署

开发环境由 Vite 将 `/api` 代理到本地 Spring Boot。生产环境通过环境变量指定 API 基础地址。模板 Mock 默认删除；手工联调必须访问真实后端，自动测试使用 Vitest mock 隔离网络。

空仪表盘只展示后台已连接、当前账号、角色和基础欢迎信息，不伪造尚未对接的统计数据。

## 10. 测试与验收

单元测试至少覆盖：

- 系统语言映射、语言持久化及中文回退
- token 保存、恢复、原子替换和清理
- 多请求并发时只执行一次 refresh
- refresh 失败后清理会话并返回登录页
- 原请求最多重放一次，refresh 请求不递归重试
- ADMIN/DEMO 路由与只读权限
- API 业务码到前端错误类型的映射

浏览器联调至少覆盖：

- ADMIN 登录、刷新页面、访问仪表盘和退出
- DEMO 登录及全局只读状态
- access token 过期后的自动刷新
- refresh token 失效后返回登录页
- 中文、日文、英文切换及刷新后保持

阶段结束执行：

```text
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

同时运行后端认证相关测试，阶段最终运行 `mvn clean test`。Docker 不可用时允许按现有条件跳过 Testcontainers MySQL 测试，但必须记录跳过数量和原因。
