## 前台原始语义

旧前台文章模型中存在两个展示属性：

- `pinned`：置顶文章。
- `feature`：推荐文章。

首页首屏不是普通文章列表的简单前三篇，而是一个带业务语义的展示区：

- 第一位是 1 篇置顶大卡片。
- 第二位是最多 2 篇推荐小卡片。
- 后续区域继续展示普通文章列表。

因此，本轮迁移不能让前台长期用 `records[0]` 或 `records.slice(...)` 从普通分页中推断置顶/推荐语义。

## 已定方向

### 1. 使用独立首页展示槽位

置顶和推荐不并入文章发布状态。文章状态继续表达发布生命周期，例如 `DRAFT`、`PUBLISHED`、`PASSWORD`、`SCHEDULED` 等。

首页展示使用独立槽位字段表达，建议枚举：

```text
NONE
PINNED
FEATURED
```

含义：

- `NONE`：普通文章，不进入首页置顶/推荐展示区。
- `PINNED`：首页置顶大卡片。
- `FEATURED`：首页推荐小卡片。

### 2. 后端兜底槽位约束

槽位数量限制由后端保证，不能只依赖后台前端校验。

- `PINNED` 最多 1 篇。
- `FEATURED` 最多 2 篇。
- 只有 `PUBLISHED` 文章允许设置为 `PINNED` 或 `FEATURED`。
- `DRAFT`、`PASSWORD`、`SCHEDULED` 等非普通公开文章不得进入首页槽位。
- 超过数量限制时拒绝保存，返回明确业务错误。
- 不自动替换旧文章，避免后台误操作导致首页内容被静默挤掉。

如果文章从 `PUBLISHED` 改为非公开状态，建议后端自动清空首页槽位为 `NONE`。这样可以避免后台用户先手动取消槽位再改状态的额外操作，也能保证前台不会展示不可公开访问的文章。

### 3. 首页公开接口返回明确结构

首页接口应由后端聚合首页所需数据，不让前台继续从普通文章分页中切片推断。

建议响应结构：

```json
{
  "pinnedArticle": null,
  "featuredArticles": [],
  "articles": []
}
```

字段语义：

- `pinnedArticle`：可为空，最多 1 篇。
- `featuredArticles`：最多 2 篇。
- `articles`：普通文章分页或列表，按 `publish_at DESC, id DESC` 排序。

`articles` 应排除已经进入 `PINNED` 或 `FEATURED` 的文章，避免同一篇文章在首页重复出现。

### 4. 前台展示不伪造槽位语义

前台按后端返回的实际槽位数据渲染：

- `Feature` 是首页首屏大卡片区域，内部使用 `HorizontalArticle` 展示主文章。
- `FeatureList` 是首页首屏次级推荐区域，内部展示 `home.recommended` / `EDITOR'S SELECTION` 标题块和若干 `ArticleCard`。
- 有 `pinnedArticle` 时，`Feature` 主卡片使用置顶文章，并允许显示置顶语义标识。
- 没有 `pinnedArticle` 时，主卡片区域可以由普通文章自然补位，但不能显示置顶语义标识。
- `featuredArticles` 有 1 篇就显示 1 篇推荐文章，有 2 篇就显示 2 篇推荐文章，并允许显示推荐语义标识。
- `featuredArticles` 为空时，推荐语义标题块和推荐标识不显示；推荐卡片区域可以由普通文章自然补位，但不能显示推荐语义标识。

换句话说，首页卡片布局区域可以不留空坑位，普通文章可以自然上移填充视觉位置；但 `PINNED` / `FEATURED` 这类业务标签、标题块和标识只在真实设置了槽位时出现，不能把普通文章包装成置顶或推荐。

### 5. 后台编辑入口和提示

后台文章新建/编辑页需要提供首页展示槽位选择：

- 普通文章。
- 置顶。
- 推荐。

后台页面需要明确提示：

- 置顶最多 1 篇。
- 推荐最多 2 篇。
- 只有已发布文章可以进入首页展示槽位。
- 超过限制时保存会失败，不会自动替换已有置顶或推荐文章。

当文章当前不是 `PUBLISHED`，槽位选择应禁用或只能选择普通文章。若用户在同一次提交中把文章发布为 `PUBLISHED` 并选择槽位，可以允许提交，但最终校验仍以后端为准。

## 三端变更摘要

### 后端

- 在 `t_article` 新增独立首页展示槽位字段 `homepage_slot`，不并入文章状态。
- 增加 `PINNED` 最多 1 篇、`FEATURED` 最多 2 篇、仅 `PUBLISHED` 可设置槽位的业务校验。
- 首页公开接口返回 `pinnedArticle`、`featuredArticles`、`articles`。
- 普通文章列表继续按 `publish_at DESC, id DESC` 排序，并排除已进入首页槽位的文章。
- 超限保存拒绝，不自动替换。

### 后台管理端

- 文章新建/编辑页增加首页展示槽位选择。
- 非 `PUBLISHED` 文章禁用槽位或只能选择普通文章。
- 展示置顶/推荐数量限制提示。
- 处理后端返回的槽位超限错误。

### 前台展示端

- 首页改为消费后端首页聚合接口。
- `Feature` / `FeatureList` 这类首屏卡片区域可以继续维持页面视觉密度。
- 有真实 `PINNED` 才显示置顶语义标识。
- 有真实 `FEATURED` 才显示推荐标题块或推荐语义标识。
- 普通文章可以自然补位，但不伪装成置顶或推荐。

## 影响面

### 后端 content

- 文章 schema 需要新增 `homepage_slot` 字段。
- Article Entity / Domain / Application / Web DTO 需要补充槽位字段。
- 文章保存、更新、状态流转需要补槽位合法性校验。
- 公开首页查询需要聚合置顶、推荐和普通文章。

### 后台 admin

- 文章表单增加槽位选择。
- 文章列表可考虑显示当前槽位，方便站长识别首页内容。
- 表单提交需要处理超限错误和非公开状态限制。

### 前台 blog

- 首页不再用普通分页切片生成置顶/推荐区。
- 首页根据 `pinnedArticle` 和 `featuredArticles` 是否存在决定是否渲染置顶/推荐语义标识。
- `Feature` / `FeatureList` 卡片区域是否保留，需要按页面视觉继续实现；普通文章可以补位，但不能带置顶/推荐语义。
- 普通文章列表只渲染后端返回的 `articles`。

### API 文档和测试

- 公开首页接口需要文档化响应结构。
- 后台文章保存/更新接口需要文档化 `homepage_slot` 字段和错误语义。
- 后端测试至少覆盖：
  - `PINNED` 超过 1 篇时拒绝。
  - `FEATURED` 超过 2 篇时拒绝。
  - 非 `PUBLISHED` 文章设置槽位时拒绝或被清空。
  - 首页普通文章列表排除已置顶/推荐文章。

## 待实现时继续确认

1. 槽位文章之间是否需要人工排序。本轮按固定规则处理：`PINNED` 只有 1 篇，`FEATURED` 最多 2 篇且可按发布时间倒序。
2. 文章从 `PUBLISHED` 改为非公开时，后端自动清空 `homepage_slot` 为 `NONE`。
