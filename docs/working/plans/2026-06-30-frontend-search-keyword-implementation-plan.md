# 前台搜索 keyword 接入实施思路

> 状态：实施思路
> 适用范围：前台 blog、后端 content、公开 API
> 最后校准：2026-06-30
> 权威程度：过程计划，裁决后需提炼到 handbook

## 本文档回答什么问题

本文档记录已经裁决的前台搜索迁移方向，作为后续拆分正式实现任务前的实施依据。

本文档不直接替代 `docs/handbook/start-here/open-issues.md`、`docs/handbook/api/article.md` 或业务规则文档。实现完成后，需要把最终结论提炼回 handbook。

## 前台原始语义

Aurora/Hexo 原始搜索是弹窗即时搜索：

- 打开搜索弹窗时加载 `/search.json`。
- 浏览器内基于本地索引搜索。
- 搜索范围是 `title` 和 `content`。
- 命中结果会截取 content 片段并用 `<mark>` 高亮。
- 支持最近搜索缓存。
- 支持键盘上下选择和回车跳转。
- 点击结果跳转文章 slug 路由。

该体验依赖静态搜索索引。V2 后端当前没有 search index，也没有独立全文搜索接口。

## 已定方向

### 1. 第一版使用后端 keyword 查询

第一版不做独立全文搜索系统，也不生成 `/search.json` 静态索引。

前台搜索弹窗保留，但数据源改为后端公开文章 keyword 查询：

```text
GET /api/public/articles?keyword={keyword}&page=1&size=...
```

### 2. 搜索范围限定为标题和摘要

第一版搜索范围限定为标题和摘要。

不搜索正文，原因：

- 当前后端公开文章列表 keyword 查询只覆盖标题和摘要。
- 直接对正文做 SQL LIKE 性能和体验都不可控。
- PASSWORD 文章正文不应通过 search index 或搜索片段泄露。
- 如果后续需要正文搜索，应单独设计 search index 或全文搜索服务。

### 3. 不做高亮片段

第一版不做服务端高亮片段。

搜索结果展示标题和摘要即可。前台可以保留弹窗样式，但不承诺像原本本地索引一样展示命中正文片段。

### 4. 公开口径与公开文章列表一致

搜索结果使用公开文章列表口径：

- 文章未软删除。
- 状态为 `PUBLISHED` 或 `PASSWORD`。
- `publish_at <= now`。

`PASSWORD` 文章可以出现在搜索结果中，但只展示公开元数据和摘要，不返回正文。

### 5. 保留弹窗交互

前台尽量保留原搜索弹窗交互：

- 搜索弹窗入口。
- 最近搜索。
- 键盘上下选择。
- 回车跳转。
- 点击结果跳转文章 slug 路由。

数据来源和展示内容变化，但不主动取消弹窗体验。

### 6. 后续增强另行设计

如果后续需要更强搜索，再单独裁决：

- 公开 search index API。
- 构建期或运行时搜索索引。
- 正文搜索。
- 搜索结果高亮。
- 专用全文搜索引擎。

这些不进入第一版。

## 三端变更摘要

### 后端

- 第一版不新增独立搜索接口。
- 继续使用 `/api/public/articles?keyword=...`。
- 确认 keyword 查询范围为标题和摘要。
- 确认公开口径为未软删除、`PUBLISHED/PASSWORD`、`publish_at <= now`。
- 不返回正文、不返回搜索高亮片段。
- 后端测试应覆盖 keyword 命中标题、命中摘要、不命中正文、过滤非公开文章。

### 后台管理端

- 当前后台管理端不需要修改。
- 搜索迁移只影响公开读者端，不影响后台文章管理搜索。

### 前台展示端

- `SearchModal` 不再加载旧 `/search.json`。
- 搜索输入改为调用 V2 公开文章 keyword 查询。
- 搜索结果展示标题和摘要。
- 点击结果跳转文章 slug 路由。
- 最近搜索和键盘导航尽量保留。
- 原本基于 content 的本地高亮逻辑需要移除或降级。

## 影响面

### 后端 content

- 公开文章列表接口已有 keyword 参数，需确认测试覆盖搜索口径。
- 若现有 VO 字段足够前台展示，可不新增后端字段。
- 若前台弹窗需要更小响应，可后续再设计轻量 search VO，本轮不默认新增。

### 前台 blog

- 搜索 store 不再维护 `SearchIndexes` 本地索引作为主要数据源。
- 搜索弹窗需要处理远程请求状态、空结果和错误状态。
- 需要防抖，避免每次输入都立即发请求。
- 最近搜索缓存保存文章标题、slug 和摘要即可。
- 文章跳转使用 O-014 的 slug 主导公开路由。

### API 文档

- `docs/handbook/api/article.md` 需要明确 keyword 搜索范围是标题和摘要。
- 文档需要明确 keyword 搜索不搜索正文。
- 文档需要明确搜索结果公开口径。

## 待实现时继续确认

1. 搜索弹窗请求防抖时间。本轮建议 250-300ms。
2. 搜索结果数量。本轮建议弹窗请求第一页，size 取 8-12。
3. keyword 最小长度。本轮建议空字符串不请求，1 个字符是否请求由前台体验决定。
4. 搜索错误状态文案是否复用文章列表错误文案。

## 建议拆分

### P1：API 契约和测试口径

- 在 handbook 中明确公开文章 keyword 搜索范围。
- 补后端 keyword 搜索测试，覆盖标题、摘要、正文和公开状态过滤。

### P2：前台搜索数据源迁移

- 新增或复用公开文章 keyword 查询 client。
- 搜索弹窗改为远程查询。
- 保留最近搜索和键盘导航。

### P3：前台交互收口

- 移除 `/search.json` 依赖。
- 移除或降级本地 content 高亮逻辑。
- 调整空结果、加载和错误状态。

### P4：文档回填

- 更新公开文章 API 文档。
- 同步前台接入状态文档。
