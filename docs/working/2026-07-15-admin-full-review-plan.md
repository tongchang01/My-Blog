# 管理端全量审查与交互改造计划

> 状态：产品方向已确认；DEMO 首次登录说明、项目页脚、仪表盘首屏、表单必填标记、各业务写操作成功反馈，以及构建欢迎语、未使用弹窗、表格工具、存储与指令清理已实施；其余上游模板遗留与视觉收敛待单独批次。
> 审查基线：`feature/admin-consumption-alignment` 的 `16aaa8a`
> 范围：`frontend/apps/admin/` 与 `MyBlog-springboot-v2/` 的管理端契约、表单、反馈、交互和上游模板遗留
> 约束：本轮审查与分批实施均不改后端业务代码、不新建接口、不替代后端校验；前端只复用既有接口、校验与 `message()`。

## 1. 已确认的事实

1. 后端管理接口已有对应的前端页面入口；不应为了“接口覆盖率”新增详情页或伪功能。
2. 当前分支已实际消费文章列表的分类、标签、创建/发布时间筛选，以及统计日期范围；友链列表请求已只保留后端支持的分页参数。原有 `backend-consumption-audit.md` 中把这三项列为待做的描述已过时，后续应在同一文档中回填验证结果。
3. 业务表单多使用自定义 `validate*Form()` 加 `el-form-item :error` 显示错误，而不是 Element Plus 的 `rules`。已按该既有校验器补齐真实必填项的星号，条件必填仍由页面状态表达。
4. 现有 `src/utils/message.ts` 已封装 Element Plus `ElMessage`。文章、分类、标签、评论、友链、附件、站点配置和资料的写操作已在服务端成功后复用该入口反馈；改密保留原有成功提示。
5. 删除、隐藏、恢复等高风险动作继续使用 `ElMessageBox.confirm`，页面保留加载、禁用与错误 `ElAlert`；成功反馈已补齐，不再造第二套通知组件。
6. 上游模板遗留至少包括：`build/info.ts` 的 pure-admin 欢迎语与打包彩色输出、页脚 pure-admin 链接、全局系统设置/布局切换、全局搜索、通知中心、多标签页、`ReDialog`、`RePureTableBar`、自定义指令和若干仅由模板功能使用的依赖。它们不能一次性删除，必须先按运行引用链确认。

## 2. 审查原则

- **契约优先**：以 controller、request support、服务层规则和当前前端请求为准；已有接口只补真实有价值的入口或解释，不发明参数。
- **同一事实只维护一次**：字段规则继续由各模块现有 `validate*Form()` 保存；星号只由页面的 `required`/动态 `:required` 表达，不复制第二套校验器。
- **反馈按操作级别分层**：字段错误留在表单项；可恢复请求失败留在页面 `ElAlert`；用户主动发起且服务端成功的写操作显示短暂 `ElMessage`；不可逆或影响公开状态的动作先用 `ElMessageBox.confirm`。
- **先正确，后视觉**：先完成后端参数、状态机、权限和必填语义，再统一布局、间距、层级、空状态与动效。
- **删除以可证明未使用为前提**：全局注册不等于产品依赖。删除前检查模板、路由、store、动态调用与构建产物，并以小批次回归。

## 3. 表单必填标记计划

| 页面/动作 | 与后端对齐的必填项 | 星号策略 | 备注 |
| --- | --- | --- | --- |

| 分类/标签 | 中文名称、slug | 静态 `required` | 排序值已有默认值与范围校验，是否标星在完整契约核验后决定。 |

## 4. 操作反馈计划

### 4.1 统一保留的现有能力

- `message()`：继续作为唯一的成功/轻量失败 toast 入口，底层就是 Element Plus `ElMessage`。
- `ElMessageBox.confirm`：继续用于删除、软删除、隐藏、显隐切换、恢复、批量排序等需要确认的写操作。
- 页面 `ElAlert`：继续显示加载失败、权限不足、校验冲突等不会因两秒 toast 而消失的错误。

### 4.2 已补齐的成功反馈

仅在服务端写操作成功后提示；查询、分页、自动刷新不提示。

| 模块 | 成功反馈 |
| --- | --- |
| 文章与回收站 | 保存、删除、恢复 |
| 分类与标签 | 新建、保存、删除、分类排序保存 |
| 评论 | 通过、隐藏、回复、删除、恢复 |
| 友链 | 新建、保存、显示/隐藏、排序保存、删除 |
| 附件 | 上传完成（明确“上传或复用已有附件”）、删除、恢复、复制 URL |
| 站点配置与资料 | 保存成功（已实现） |
| 密码 | 保留现有“成功后需重新登录”提示 |

错误文案不能把后端语义抹平：例如“恢复软删除记录”与“通过隐藏评论”是不同动作；友链的隐藏不等于删除；文章首页槽位冲突必须保持表单并提示可行动原因。

## 7.1 DEMO 首次登录说明弹窗（已确认需求）

### 触发与呈现

- 仅当登录页通过 `sessionService.signIn()` 成功拿到 `type = DEMO` 的当前用户时触发。
- 先完成路由初始化并进入仪表盘，再立即显示 Element Plus `ElMessageBox.alert`；DEMO 不再额外显示普通“登录成功” toast，确保用户首先看到说明。
- 会话恢复、刷新页面、普通路由跳转不重复弹出；每次**主动重新登录** DEMO 账号都会重新说明一次。
- 使用确定按钮“我知道了”，不提供“本次不再提示”，避免演示账号被误认为可写账号。

### 弹窗文案要点

标题：`演示账号（只读）`

1. 当前登录的是 DEMO 演示账号，仅用于浏览后台能力与数据展示。
2. 所有创建、编辑、删除、排序、上传、回复和修改密码等写操作均被前端禁用，后端也会再次拒绝。
3. 为保护真实数据，非公开文章正文、评论邮箱/IP/User-Agent，以及附件内部存储信息不会向 DEMO 返回。
4. 如需执行管理操作，请使用 ADMIN 账号；弹窗不展示、也不推断任何管理员凭据。

### 保留的页面级反馈

- 继续保留各页面已有的只读禁用、操作列隐藏和局部 `ElAlert`，因为它们解释的是当前页面不能执行什么。
- 首次登录弹窗只负责解释“为什么整个账号不可写”，不替代后端 `403`、不在前端伪造权限。

### 验收

1. DEMO 主动登录后，仪表盘首次渲染即出现说明弹窗；ADMIN 登录不出现。
2. 关闭弹窗后，DEMO 仍可浏览被授权的只读页面，所有写入口保持禁用或隐藏。
3. 刷新 DEMO 会话或后台内部跳转不重复弹出；退出并重新登录 DEMO 后会再次出现。
4. 已为 `demo-notice` 弹窗 helper 增加单元测试，验证必须确认、不可点遮罩或按 Esc 关闭的呈现约束；后端权限测试保持现有 `DEMO` 写请求 `403` 断言。
