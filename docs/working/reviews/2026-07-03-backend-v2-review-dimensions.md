# 后端 V2 代码审查维度与优先级

> 状态：初稿
> 适用范围：`MyBlog-springboot-v2/` 全量
> 最后校准：2026-07-03
> 权威程度：整理过程材料

## 本文档回答什么问题

回答两件事：

1. 对本项目的 V2 后端做代码审查，一般要覆盖哪些维度、按什么优先级组织。
2. 本轮已经完成的三份 review 与后面还没盘的部分，什么时候真正开始改。

**结论先说：本轮所有 review 结果——包括已经完成的三份和后面待覆盖的维度——都统一延后到 V2 第一版部署到生产、CD 流程走通并稳定观察一段时间后再启动整改。**

原因：接下来要先测试 CI/CD 交付流程本身。此时如果把内部代码审查发现的整改混进来，会出现两类风险：

- 无法分辨"上线问题"是 CD 变更引起还是代码整改引起。
- 内部重构和上线首版并行，回滚成本变高。

因此本文档同时充当"审查 backlog"，不作为立即执行的清单。

## 审查维度与优先级

按"错了后果多严重、多难恢复"排序。上层的问题风险越高，越应先看；下层的越是"错了能慢慢修"。

### 一档：正确性与安全

一处错就要立刻处理。

1. **业务逻辑正确性** — 语义、边界、幂等、并发下的一致性。（本轮已由你排除，不在本次范围。）
2. **认证 / 授权 / 密码 / 令牌 / 会话** — 权限绕过、密钥泄漏、令牌重放、账号枚举、密码算法强度。
3. **注入面** — SQL、命令、路径遍历、SSRF、模板注入、日志注入、反序列化。
4. **并发与线程安全** — 共享可变状态、`@Scheduled` 幂等、单例服务的实例字段。
5. **事务边界与数据一致性** — `@Transactional` 传播、只读事务、跨资源（DB + 文件 / HTTP）的补偿窗口。
6. **资源管理** — try-with-resources、流关闭、临时文件生命周期、连接池、内存 / 句柄泄漏。
7. **失败路径** — 抛什么、吞什么、日志级别、5xx 是否泄漏堆栈、外部依赖超时 / 重试。

### 二档：生产就绪

上线后会咬人。

8. **可观测性** — 日志结构、trace / correlation ID、metrics、关键路径覆盖度、日志中的 PII。
9. **配置管理** — 默认值、启动期校验、profile 区分、密钥外化、`@ConfigurationProperties` 校验。
10. **性能与查询模式** — MyBatis N+1、缺索引、分页 offset 深度、大对象内存驻留、缓存策略与失效。
11. **多实例就绪** — 有状态字段（in-memory 限流、缓存）、`@Scheduled` 抢占、粘性 session。
12. **API / HTTP 语义** — 幂等方法、状态码、错误码稳定性、Content-Type、CORS、cacheability、版本化。
13. **依赖与 CVE** — Spring Boot / Netty / Jackson 等新爆漏洞、传递依赖、许可证。

### 三档：可维护性

短期没事，会拖慢后续。

14. **模块边界与依赖方向** — DDD 分层、ArchUnit 规则、包私有 vs public 泄漏。
15. **重复代码 / 抽象合理性** — 三次以上重复才抽、避免过早抽象。
16. **命名 / 风格 / 约定一致性** — 日志字段、异常变量名、错误消息、注解风格。
17. **复杂度控制** — 方法 / 类长度、圈复杂度、深嵌套。
18. **文档现状** — README、handbook 与代码 / API 的漂移。

### 四档：可演进性与团队卫生

长期问题。

19. **测试覆盖与质量** — 关键路径覆盖、集成 vs 单元比例、脆弱测试、fixture 复用、mock 层次。
20. **接口稳定性与兼容策略** — 契约测试、deprecation、feature flag 生命周期。
21. **提交与分支纪律** — 提交粒度、消息、分支策略（本仓库已有 [AGENTS.md](../../AGENTS.md) 规约）。
22. **构建 / CI** — 构建复现、缓存、集成测试环境、依赖锁定。

## 延后策略

**统一延后到 V2 第一版部署到生产环境、CD 流程完整走通、稳定观察一段时间之后再启动整改。**

在此之前的行动限制：

- 不基于三份 review 的现象开新的整改 PR。
- 不基于本文档的"待覆盖维度"追加新的调查文档。
- 遇到本次 review 里已经登记过的现象，允许"顺手改"限于同一次业务变更的正常范围内，不单独拉整改分支。
- 若在测试 CD 期间发现的是新的、未登记的问题（尤其一档：安全 / 数据一致性 / 资源泄漏），照常处理，不受本策略约束；同时补登记到本文档或对应 review 文档。

## 恢复条件

同时满足以下条件视为可以启动整改：

- V2 第一版已部署到生产环境。
- CD 流程完成至少一轮完整的"构建 → 部署 → 回滚演练"验证。
- 生产观察窗口内没有一档（安全 / 数据一致性 / 资源泄漏）新问题。

启动时的第一步不是直接改代码，而是：

- 走访三份 review 中列出的所有"待调查问题"，逐项裁决"有意 / 欠账"。
- 把"欠账"部分合并成整改计划文档，落到 [docs/working/plans/](../plans/)。
- 按 [AGENTS.md](../../AGENTS.md) 的拆分要求组织分支和提交。
