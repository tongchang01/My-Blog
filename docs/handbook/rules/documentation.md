# 文档维护规则

> 状态：当前有效
> 适用范围：全项目文档
> 最后校准：2026-07-10
> 对应代码：`docs/`、`README.md`、`AGENTS.md`
> 权威程度：文档治理规则

## 1. 目标与受众

文档必须同时服务开发人员和 AI 协作流程，重点回答当前事实、约束、操作方式和未解决事项。历史过程、会议式记录和一次性推理不进入长期文档。

正文默认使用中文，代码标识、协议名和技术专有名词保持原文。`docs/showcase/` 允许中文、英文和日文三个对外版本。

## 2. 目录

`docs/` 只保留以下内容：

```text
docs/
├── README.md       文档总入口
├── handbook/       当前开发与部署权威说明
├── governance/     仓库协作规则
└── showcase/       对外项目展示
```

应用代码目录只允许保留简短 README 和必要的上游来源说明。产品、架构、API、运维、计划和评审内容统一收口到 `docs/` 或 Git 历史。

## 3. 事实与权威源

1. 当前代码、数据库迁移、配置和可执行测试是实现事实源。
2. `handbook/` 是实现事实的人类可读说明，不得与代码并列维护第二套模型。
3. ADR 记录仍影响当前实现的决策理由；ADR 与代码冲突时必须修正实现、更新 ADR 状态或登记遗留项。
4. `governance/` 只管理仓库协作，不承载业务设计。
5. `showcase/` 只描述已经实现且适合公开介绍的能力。

## 4. 主题唯一归属

| 主题 | 唯一位置 |
| --- | --- |
| 当前进度与验证状态 | `start-here/current-status.md` |
| 未来实施顺序 | `start-here/roadmap.md` |
| 未解决事项 | `start-here/open-issues.md` |
| 术语 | `start-here/glossary.md` |
| 架构事实 | `architecture/` |
| 架构决策理由 | `adr/` |
| 开发约束 | `rules/` |
| HTTP 契约 | `api/` |
| 稳定业务语义 | `product/` |
| 前端实现边界 | `frontend/` |
| 环境、测试与部署 | `ops/` |
| 可重复开发流程 | `workflows/` |

其他文档只能摘要并链接权威位置，不能复制大段内容。

## 5. 文档头

`handbook/` 和 `governance/` 中的 Markdown 文件必须在一级标题后提供：

```md
> 状态：当前有效 / 待实现 / 已替代
> 适用范围：明确模块或流程
> 最后校准：YYYY-MM-DD
> 对应代码：一个或多个实际路径；目录索引可写“无”
> 权威程度：权威说明 / 契约摘要 / 规则 / ADR / 目录入口
```

只有完成代码核对的文档可以标记“当前有效”。“待实现”只能用于路线图和遗留事项，不能伪装成当前能力。“已替代”只用于仍有必要保留的 ADR，并必须链接替代决策。

## 6. 命名与写法

1. 开发文档使用英文小写和短横线文件名；ADR 使用四位编号。
2. `README.md` 只作为目录入口，不承载完整规格。
3. 叙述主体使用“MyBlog”“项目”“模块”“接口”“开发流程”等明确名称。
4. 长期文档不使用对话式人称代词，不出现具体 AI 工具品牌。
5. 结论使用当前时态；日期只表示最后校准时间，不用阶段日期代替状态。
6. 命令必须说明执行目录，路径必须是仓库相对路径，不能写本机绝对路径。
7. 未确认信息必须放入遗留事项，不能用模糊措辞伪装成事实。

## 7. 生命周期

1. 任务计划和评审材料只在执行期使用，不作为长期权威文档。
2. 任务完成后，有效结论写入对应权威文档，未完成项写入 `open-issues.md`，过程材料删除。
3. 已关闭遗留项从 `open-issues.md` 删除，关闭原因由提交记录追溯。
4. 失效文档、重复副本和已完成迁移说明直接删除，不建立仓库内归档。
5. 仍影响当前实现的决策理由保留为 ADR；完全失效且无当前解释价值的 ADR 删除。

## 8. 更新触发器

| 变更 | 必须同步更新 |
| --- | --- |
| Controller、请求或响应模型、错误码 | `api/` |
| 模块边界、依赖方向、数据流 | `architecture/`，必要时更新 ADR |
| 数据表、索引、迁移策略 | `architecture/schema-design.md` 与相关 ADR |
| 业务规则 | `product/` |
| blog 或 admin 页面、路由、数据源 | `frontend/` 与 `current-status.md` |
| 环境变量、启动、测试、CI、部署 | `ops/` |
| 发布范围或优先级 | `roadmap.md` 与 `open-issues.md` |

## 9. 提交前检查

1. 文档链接指向实际存在的文件。
2. 状态头完整，代码路径存在，校准日期为本次核对日期。
3. 没有旧目录、本机绝对路径、已删除接口或已关闭待办。
4. 同一事实只有一个权威位置。
5. 文档变更与对应代码变更位于同一目的提交；大规模文档治理必须单独提交。
6. `git diff --stat` 和 `git status --short` 与任务范围一致。
