# 代码注释规范

> 状态：当前有效
> 适用范围：MyBlog V2 Java 代码、测试、配置和 OpenAPI 描述
> 最后校准：2026-06-29
> 对应代码：`MyBlog-springboot-v2/src/main/java/`、`MyBlog-springboot-v2/src/test/java/`
> 权威程度：规则

## 本文档回答什么问题

本文档规定 V2 代码注释应该使用什么语言、什么时候必须写注释、什么时候不要写注释，以及 Javadoc、行内注释、OpenAPI 描述和测试注释如何分工。

## 1. 语言规则

开发代码中的注释统一使用中文。

允许保留英文的内容：

- 技术名词：Spring、MyBatis-Plus、JWT、CORS、Flyway。
- 类名、方法名、包名、配置 key。
- HTTP header、claim、错误码、数据库字段名。
- 第三方库原始概念。

禁止：

- 中英文混杂导致语义不清。
- 为了翻译技术名词而制造不常用中文说法。
- 复制旧英文模板注释作为当前业务说明。

## 2. 核心原则

注释解释“为什么”和“业务边界”，不要逐句翻译代码。

反例：

```java
// 如果用户为空
if (user == null) {
}
```

正例：

```java
// 直连请求不信任代理头，避免客户端伪造登录审计 IP。
if (!isTrustedProxy(remoteAddress)) {
    return remoteAddress;
}
```

## 3. 必须写注释的场景

| 场景 | 注释应说明什么 |
|------|----------------|
| 公开 HTTP operation | `@Operation.summary` 描述调用方看到的业务动作 |
| 高风险 request/response 字段 | `@Schema` 说明权限、状态、时间、nullable、枚举、敏感语义 |
| 权限边界 | 为什么 ADMIN/DEMO/GUEST 被允许或拒绝 |
| 状态流转 | 当前状态、目标状态、拒绝原因 |
| 软删除/恢复 | 公开可见性、引用保护、审计语义 |
| 登录、token、密码 | 失败策略、撤销策略、敏感信息处理 |
| 限流、锁、并发 | key、阈值、行锁、幂等或回滚原因 |
| 复杂 SQL | 筛选、聚合、排序和锁定口径 |
| 上传和内容清洗 | 信任边界、真实格式识别、XSS 防护原因 |
| 配置项 | 默认值、生产风险和缺失时行为 |

## 4. Javadoc 规则

类级 Javadoc 应说明该类的业务职责或技术边界。

方法级 Javadoc 适合：

- public API。
- application service 的关键用例。
- 端口接口。
- 不容易从命名看出语义的方法。

普通 getter/setter、简单构造器、无业务语义的私有方法不强制写 Javadoc。

## 5. 行内注释规则

行内注释用于解释局部决策。

必须考虑行内注释的场景：

- 事务提交前后顺序。
- 行锁和并发收敛。
- token 撤销和版本比较。
- trusted proxy 和 IP 解析。
- PASSWORD 文章、DEMO 裁剪等特殊边界。
- H2/MySQL 兼容写法。

禁止把每一行代码翻译成中文。

## 6. OpenAPI 注释规则

- Controller 类使用 `@Tag` 描述业务分组。
- 每个公开 HTTP operation 必须有非空 `@Operation.summary`。
- 权限、状态、时间、nullable、枚举和敏感字段用 `@Schema` 明确契约。
- 普通自解释字段不机械补注解。
- OpenAPI 描述不得与 API 契约文档冲突。

## 7. SQL 注释规则

复杂 XML SQL 必须写中文注释。

应说明：

- 服务哪个接口或业务场景。
- 哪些状态会被包含或排除。
- 聚合口径。
- 排序理由。
- `for update` 等锁定语义。

简单单表 CRUD 不需要为了注释而注释。

## 8. 测试注释规则

普通测试不强制写注释。

以下测试建议写场景说明：

- 权限边界。
- 状态流转。
- 安全规则。
- 并发和事务回滚。
- 容易误删或误展示数据的场景。
- H2 与 MySQL 方言兼容场景。

示例：

```java
/**
 * 验证已删除评论不会出现在前台列表中。
 * <p>该场景保护后台软删除语义，避免读者继续看到已处理评论。</p>
 */
@Test
void excludesDeletedCommentsFromPublicList() {
}
```

## 9. 配置注释规则

新增配置项时应说明：

- 用途。
- 默认值。
- local/test/prod 差异。
- 生产风险。
- 缺失时是否启动失败。

密钥、密码和 token 相关配置不得在注释中给出真实值。

## 10. 不需要注释的场景

可以不写注释：

- 命名已经足够清楚的简单 DTO 字段。
- 纯机械映射。
- getter/setter。
- 简单构造器。
- 没有业务语义的测试准备样板。
- 只为满足数量而写的重复性注释。

## 11. 禁止事项

- 禁止用注释重复代码。
- 禁止写与代码行为不一致的注释。
- 禁止在注释里保留过期 TODO。
- 禁止在注释中暴露密钥、密码、token、真实账号或生产连接信息。
- 禁止让 `@TableField`、`@TableName` 这类映射注解替代业务注释。

## 12. 评审口径

评审注释时看三点：

1. 是否解释了命名无法表达的业务边界。
2. 是否帮助维护者避免误改安全、权限、状态和事务语义。
3. 是否与当前代码、API 契约和文档一致。

注释数量不是目标，信息质量才是目标。
