# 写一份 ADR（SOP）

> 目标：当做出影响架构的决策时，正确写下一份 ADR。

## 1. 什么决策需要写 ADR

写：
- 选型决策（换 ORM、加新依赖、升级框架大版本）
- 架构边界变化（新增模块、合并模块、改层结构）
- 安全策略（密钥管理、鉴权方式）
- 数据库策略（迁移工具、分库、缓存引入）
- 跨模块通信方式变更

不写：
- bug 修复
- 小型重构
- 单纯的代码风格调整（改 `rules/` 即可）

## 2. 步骤

### 步骤 1：领下一个编号

`../adr/` 目录下文件以 `NNNN-...` 命名，连号不空号。看最后一个号 + 1。

### 步骤 2：用模板写 `NNNN-短主题.md`

模板见 `../adr/README.md`，至少含：
- 状态、日期、决策者
- 背景
- 备选方案（至少 2 个，含被否的）
- 决定
- 理由（关键权衡）
- 后果（正面 + 负面 + 后续关注）
- 相关（相关 rules / 相关 ADR / 取代关系）

### 步骤 3：更新相关 rules

ADR 落地后，把"怎么做"沉淀到 `../rules/`。
ADR 是"为什么"，rule 是"怎么做"，**两者并存**。

### 步骤 4：更新 arch

若架构现状有变，同步 `../architecture/` 对应文件。

### 步骤 5：取代旧 ADR（如有）

若新决策取代旧决策：
- **不**删除旧 ADR
- 旧 ADR 状态改为 `superseded by ADR-NNNN`
- 新 ADR 在"相关"中标注 `取代 ADR-XXXX`

## 3. 写作要点

- 背景：写当时**事实**，不写"为何错"的判断
- 备选方案：被否的方案也要诚实列出，否则后人不知道考虑过什么
- 理由：解释**权衡**，不是吹某方案多好
- 后果：负面也要写，记录代价才能后续评估

## 4. ADR 不是

- ❌ 工作日志
- ❌ 项目计划
- ❌ 教程
- ❌ rule（"怎么做"）

ADR 是**当时为什么这么决**的快照，写完一般不再改（除非状态变更）。
