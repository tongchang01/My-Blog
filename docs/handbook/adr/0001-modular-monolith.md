# ADR-0001: 采用模块化单体（Modular Monolith）

- 状态：accepted
- 日期：2026-04
- 决策者：项目负责人

## 背景

V1 是典型的 Spring Boot 单体，所有业务逻辑混在一起，包结构按技术分层（controller/service/mapper），随业务增长逐渐难以维护。重构 V2 时需要决定：

- 继续单体？
- 拆微服务？
- 模块化单体？

本项目是个人博客，部署规模小、团队规模小（1 人），但代码量已上万行，存在评论、文章、用户、系统配置等明显边界。

## 备选方案

- **方案 A：传统单体**。继续按技术分层。简单，但无业务边界。
- **方案 B：微服务**。彻底拆分。技术先进，但运维代价大，与项目规模严重不匹配。
- **方案 C：模块化单体**。一个部署单元，但按业务模块隔离包结构，模块间通过明确接口通信。

## 决定

选 C：模块化单体。

## 理由

- 部署仍是单体，无需引入服务注册、网关、链路追踪等复杂度
- 内部按模块隔离，强制业务边界，未来如需拆分微服务，迁移成本低
- 配合 ArchUnit 守护规则，可在编译期防止跨模块乱依赖
- 与个人项目规模匹配

## 后果

正面：
- 业务边界清晰，新增功能定位明确
- 测试可在模块级别独立
- 为未来拆分留好接口

负面：
- 比传统单体多一层包结构成本
- 需要 ArchUnit 守护规则维护

后续需关注：
- 模块间通信若出现循环依赖，必须及时重构
- 新增模块时同步更新 ArchUnit 规则
