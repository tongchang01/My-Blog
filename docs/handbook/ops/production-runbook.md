> 最后校准：2026-07-11
> 对应文档：`deployment-direction.md`、`release-checklist.md`、`environment.md`
> 权威程度：生产操作顺序

## 私有信息说明

敏感生产信息已保存在仓库外的私有生产台账中，包括真实 IP、AWS 资源标识、安全组规则、Docker 卷标识、宿主机路径、证书状态、备份位置和环境变量值。

本手册不得写入这些真实值。执行人员必须在维护窗口前核对私有台账，并将本手册中的通用对象映射到已确认的真实资源；不得根据名称猜测服务器、容器、卷或目录。

## 执行原则

- 不重装操作系统，只清理 V1 应用层资源。
- 不迁移 V1 业务数据，V2 使用全新数据库和数据卷。
- 先获得可恢复证据，再停止或删除任何资源。
- 删除容器不等于删除数据卷；旧 MySQL 卷保留 7 天。
- 清理命令逐项执行和复核，不使用通配符批量删除。
- 未发布可运行镜像或未通过 CI 时，不得开始停机清理。

## 阶段一：完成发布产物

- [ ] 首个管理员一次性初始化能力已实现并通过测试。
- [ ] `myblog-api` Dockerfile 已验证 Java 17、非 root 用户和健康检查。
- [ ] `myblog-web` 已包含 blog/admin 生产构建和 Caddy 配置。
- [ ] 生产 Compose 已固定私有网络、持久卷、资源限制和 restart policy。
- [ ] 后端、真实 MySQL 专项、blog/admin 测试和生产构建全部通过。
- [ ] 两个公共 GHCR 镜像已使用同一提交 SHA 发布，可匿名拉取。
- [ ] 已记录待发布提交 SHA，不使用无法追溯的镜像标签。

## 阶段二：准备 AWS 与生产配置

- [ ] AWS root 已启用 MFA，日常操作使用单独管理员身份。
- [ ] EC2 IAM Role 只包含 V2 S3 附件前缀所需权限。
- [ ] EC2 已要求 IMDSv2，容器 metadata hop limit 已核对。
- [ ] 安全组目标为：80/443 公网、22 受限、不开放 3306/8080。
- [ ] 已生成全新的 MySQL、JWT、统计哈希和管理员凭据。
- [ ] V2 使用只允许访问 `myblog_v2.*` 的专用 MySQL 账号，不使用 root；账号具备 Flyway 建表、变更表和应用读写所需权限。
- [ ] 生产环境文件仅 root 可读，且未进入 shell history、Git 或日志。
- [ ] 已安排可接受停机的维护窗口。

环境变量名称和约束以 [`environment.md`](environment.md) 为准；环境变量值只从私有生产台账和受控密钥文件获取。

## 阶段三：建立回滚点

- [ ] 创建当前 EC2 的 AMI 或根 EBS 快照，记录快照标识和创建时间。
- [ ] 导出 V1 MySQL 逻辑备份，确认文件存在且非空。
- [ ] 如需保留 V1 环境文件，已离线加密保存且不会复用于 V2。
- [ ] 记录旧 MySQL 数据卷的真实标识。
- [ ] 为快照、备份和旧卷记录统一的 7 天清理日期。
- [ ] 已确认从快照恢复实例的入口和权限可用。

快照或备份未确认前，不得进入下一阶段。

## 阶段四：清理 V1 应用层

按私有台账中的真实对象逐项处理：

1. 停止 V1 Java，确认进程不再写入数据库。
2. 停止旧 Nginx、MySQL、Redis 和 RabbitMQ 容器。
3. 删除上述旧容器，但保留旧 MySQL 数据卷。
4. 归档或删除旧 JAR、旧前端目录、旧 Nginx 配置和 Certbot 状态。
5. 再次确认 80/443 已释放，旧服务不会因 restart policy 自动恢复。
6. 增加并验证 1 GiB swap。

每一步完成后立即核对实际进程、容器、端口和磁盘状态。发现对象与私有台账不一致时停止操作，不继续猜测。

## 阶段五：部署 V2

1. 安装或确认 Docker Compose 可用。
2. 放置经过版本控制的 Compose/Caddy 配置和 root-only 环境文件。
3. 使用已记录提交 SHA 拉取 `myblog-api` 与 `myblog-web`。
4. 启动 `mysql`，等待健康检查通过。
5. 启动 `api`，等待 Flyway 完成并确认 `/actuator/health` 正常。
6. 新增或确认管理端 DNS 记录。
7. 启动 `web`，由 Caddy 接管 80/443 并申请证书。
8. 检查 Compose 状态和容器日志，确认无凭证、迁移或代理错误。

通用检查命令：

```bash
docker compose pull
docker compose up -d
docker compose ps
docker compose logs --tail=200 api
docker compose logs --tail=200 web
```

实际工作目录、Compose 文件位置和环境文件位置以私有生产台账为准。

## 阶段六：生产验收

最低验收范围：

- [ ] 主域名、`www` 和管理端 HTTPS 正常。
- [ ] `/api` 前缀被完整保留，健康检查正常。
- [ ] 首个管理员创建成功；初始化开关和明文密码变量已移除。
- [ ] 登录、刷新、退出和后台权限正常。
- [ ] Flyway 版本与 checksum 正常。
- [ ] S3 上传、公开读取、删除和恢复正常。
- [ ] 客户端 IP 只信任 Compose 代理网段。
- [ ] 3306 和 8080 未对公网开放。
- [ ] 内存、swap、磁盘、日志和数据库连接稳定。

完整产品冒烟与回滚门槛继续执行 [`release-checklist.md`](release-checklist.md)，本手册不复制其内容。

## 回滚

### V2 镜像问题

- 将镜像引用恢复到上一条已验证提交 SHA。
- 重新执行 `docker compose pull` 和 `docker compose up -d`。
- 重新运行健康检查和最低生产验收。

### 数据库迁移问题

- 停止发布，不修改 Flyway history，不假定镜像回滚会自动回滚 schema。
- V2 使用全新数据卷；必要时删除失败的新卷并从空库重新部署。

### 主机或清理事故

- 停止继续修改当前实例。
- 使用已记录的 AMI/EBS 快照恢复旧环境。
- 恢复步骤和真实资源标识从私有生产台账获取。

## 七天收尾

- [ ] 连续观察 7 天，未发生需恢复 V1 的问题。
- [ ] 删除旧 MySQL 数据卷前再次核对其标识与新 V2 卷不同。
- [ ] 删除临时 AMI/EBS 快照和 V1 MySQL 保险备份。
- [ ] 清理 V1 加密环境文件副本。
- [ ] 记录实际发布 SHA、执行日期、异常和最终资源状态。
- [ ] 将确认后的非敏感事实同步回仓库文档；敏感事实继续只更新私有生产台账。
