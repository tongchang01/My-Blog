# ADR-0013: V2 不再兼容 V1 数据结构

- 状态：accepted
- 日期：2026-06
- 决策者：项目负责人
- 取代：早期 V2 重构的隐含假设"兼容 V1 数据库结构"

## 背景

早期 V2 重构假设 "V1 数据库结构保留，V2 用 Entity 字段映射 + 注释屏蔽旧库语义"。该假设带来的实际负担：

- 业务代码到处带"旧库兼容"包装层（`is_review` Integer ↔ `reviewStatus` 枚举等）
- 表名沿用原作者命名习惯（`t_about` 等），命名不规范
- 数字魔法值（`type=1/2/3/4/5` 表示评论类型）散落各处
- 字段命名风格不统一（部分驼峰、部分缩写）
- 软删除字段强制走 `is_delete` 0/1 整型

新背景：
- 项目所有者决定**三端 + 数据库全量重构**（2026-06）
- V1 仍在线上运行，但**本地重构不需要兼容**
- 数据量极小（不到 20 篇文章），数据迁移工程量可忽略

## 决定

V2 数据库 schema **完全重新设计**，不沿用 V1 表结构。V1 表仅作业务语义参考。

## 理由

- 兼容成本（代码层包装层 + 心智负担）远大于"重设计 + 一次性数据导入"成本
- 数据量小，迁移可用脚本或手工，不需要工程级 ETL
- 借机做命名规范化、字段类型规范化（布尔用 BOOLEAN 不用 TINYINT(1)、时间用 TIMESTAMP、软删除统一策略）
- V1 不下线，仍可对照查业务行为，无需保留兼容层

## 后果

正面：
- 业务代码摆脱"旧库兼容"包装层，更直观
- 表名、字段名、类型规范统一
- Entity 直接表达业务语义，不需要枚举 ↔ 数字的转换
- 早期 V2 中带"旧库兼容"注释的 Entity 在业务清空时一并清理

负面：
- 已写的 identity/content/comment 业务代码需要按新 schema 重写（架构骨架保留，业务清空）
- 需要写一次性数据导入脚本（见 `migration/v1-data-import.md`）
- 早期文档中"兼容旧库"相关段落需修正（已完成）

后续需关注：
- 新 schema 设计原则见 ADR-0014
- 新 schema 定稿见 `arch/schema-design.md`
- 业务规格梳理见 `product/`
