# 前台 V2 后端接入评估计划

> 日期：2026-06-28  
> 状态：待审阅，未提交  
> 背景：后台已基本可用，前台仍处于从外部主题仓库改造成 SPA 后的混沌状态。下一阶段不继续扩展后台，而是评估并逐步打通前台与 V2 后端。

## 1. 当前判断

后台已经能管理博客核心数据，但前台还没有系统接入后端。

现在继续补后台高级功能收益不高，风险是做出更多“后台可配置，但前台暂时不用”的能力。

下一阶段主线应该切到前台：

- 先盘点前台当前依赖哪些静态 JSON、默认配置和旧主题逻辑；
- 再把这些数据源映射到 V2 后端已有接口；
- 缺接口的地方先记录，不急着补；
- 优先跑通博客阅读链路，而不是一次性清空所有 mock / JSON。

## 2. 阶段目标

把前台从“能跑的主题 SPA”推进到“能消费 V2 后端的博客前台”。

第一阶段只追求：

1. 页面能稳定打开。
2. 核心内容来自后端。
3. 静态 JSON 的职责被标清。
4. 缺口清单明确。

不追求：

- 一次性重写前台架构；
- 一次性删除所有静态 JSON；
- 一次性接完评论、搜索、RSS、Sitemap；
- 一次性把主题所有配置后台化。

## 3. 工作原则

### 3.1 不重写

前台目前是从现成主题仓库改造来的。短期不做大拆大改。

先保留现有组件、路由、样式和布局，只替换数据来源。

### 3.2 不先删 JSON

静态 JSON 先保留。

接一个模块，替换一个模块的数据源。确认稳定后再决定是否删除对应 mock。

### 3.3 后端已有接口优先

已有后端接口的模块先接。

缺接口的模块先记录，除非阻断核心阅读链路，否则不立即补后端。

### 3.4 评论后置

评论牵涉：

- 提交；
- 审核；
- 回复；
- 站长标识；
- 头像；
- Gravatar；
- 层级展示。

它不适合作为前台接后端的第一批。

## 4. 阶段 1：数据源盘点

目标：搞清楚前台现在到底靠什么数据运行。

需要盘点这些区域：

| 区域 | 当前重点 |
|---|---|
| 首页 | 文章列表、特色区、站点配置、侧边栏 |
| 文章详情 | 文章正文、作者、上一篇/下一篇、目录、评论入口 |
| 分类页 | 分类列表、分类下文章 |
| 标签页 | 标签列表、标签下文章 |
| 归档页 | 时间线、分页 |
| 友链页 | 友链列表、头像、描述 |
| 关于页 | 作者介绍、站点介绍 |
| 作者卡片 | `authors/blog-author.json`、默认配置、社交链接 |
| 评论 | 当前是否仍使用旧第三方评论插件或 mock |
| 站点配置 | 标题、副标题、Logo、Favicon、语言、主题默认值 |

输出一张映射表：

| 模块 | 当前前台数据源 | 后端现状 | 动作 | 优先级 |
|---|---|---|---|---|
| 首页文章 | 待查 | 公开文章接口已有 | 接后端 | P0 |
| 文章详情 | 待查 | 公开文章详情已有 | 接后端 | P0 |
| 分类 / 标签 | 待查 | 公开分类标签接口已有 | 接后端 | P0 |
| 归档 | 待查 | 可能需复用文章列表 | 评估 | P1 |
| 作者卡片 | `authors/blog-author.json` | 后端有 profile，但缺聚合接口 | 后置补接口 | P1 |
| 友链 | 待查 | 公开友链接口已有 | 第二批接 | P1 |
| 关于页 | 待查 | 站点配置 about 已有 | 第二批接 | P1 |
| 评论 | 待查 | 后端评论接口已有 | 后置 | P2 |
| 搜索 | 待查 | 后端未必完整 | 后置 | P2 |

## 5. 阶段 2：核心阅读链路

优先打通博客最基本的阅读体验。

顺序：

1. 站点配置
2. 首页文章列表
3. 文章详情
4. 分类页
5. 标签页
6. 归档页

验收标准：

- 前台不依赖文章 mock 也能显示真实文章；
- 后台新建公开文章后，前台能看到；
- 后台修改标题、摘要、封面后，前台能更新；
- 分类 / 标签能从后端真实数据展示；
- 多语言路径不被破坏。

暂不处理：

- 评论；
- 作者卡片完整动态化；
- 搜索；
- RSS；
- Sitemap。

## 6. 阶段 3：侧边栏与静态页面

核心阅读链路跑通后，再处理外围展示。

顺序：

1. 作者卡片
2. 友链页
3. 关于页
4. Header / Footer 的站点信息
5. Logo / Favicon
6. 社交链接

这里要处理“作者资料”的模型差异。

当前前台作者 JSON 是聚合视图：

- `name`
- `slug`
- `avatar`
- `link`
- `description`
- `socials`
- `categories`
- `tags`
- `word_count`
- `post_list`

后端已有的资料分散在：

- `t_user_info`
- `t_site_config`
- 文章 / 分类 / 标签统计

合理做法不是让数据库字段追平 JSON，而是后端提供一个前台 author view 接口，拼出前台需要的数据。

候选接口：

```http
GET /api/public/authors/blog-author
```

短期可以先保留静态 JSON，等核心阅读链路稳定后再接。

## 7. 阶段 4：评论链路

评论最后接。

原因：

- 展示结构比文章列表复杂；
- 后台回复已经存在，但前台站长标识还没定；
- 评论头像目前更适合用 Gravatar 或作者资料识别，不适合马上改评论表。

目标效果：

1. 前台提交评论。
2. 后台审核通过。
3. 前台显示评论。
4. 后台回复。
5. 前台显示站长回复。
6. 站长回复能有清晰标识。

这一步再决定是否需要：

- `isAuthor`
- `authorRole`
- 站长徽标
- 作者资料头像复用

## 8. 阶段 5：清理静态 JSON

等后端接口稳定后再清理。

候选清理对象：

- `frontend/apps/blog/public/api/posts`
- `frontend/apps/blog/public/api/authors`
- `frontend/apps/blog/public/api/categories`
- `frontend/apps/blog/public/api/tags`
- 旧主题 mock 数据
- 不再使用的第三方评论适配

清理原则：

- 已有后端替代接口；
- 页面已验收；
- 测试或手工验收记录存在；
- 不一次性大删。

## 9. 暂不建议做的事

这些现在先不做：

- 前台整体重写；
- 主题配置全部后台化；
- 导航菜单后台化；
- 评论表加头像字段；
- 用户角色管理；
- 全量删除 mock；
- 生产部署整理。

这些不是不要，而是等前台主链路稳定后再做。

## 10. 下一步建议

如果本计划确认，下一步不是马上写业务代码，而是执行阶段 1：

1. 扫描前台路由、页面和 store。
2. 列出所有静态 JSON / mock / defaults 数据源。
3. 对照后端已有 public/admin API。
4. 输出一份更具体的“前台数据源映射表”。
5. 再决定第一批接哪个页面。

第一批实现建议从首页文章列表和文章详情开始。

理由：这是博客最核心链路，最容易验证后台文章管理是否真的有用。
