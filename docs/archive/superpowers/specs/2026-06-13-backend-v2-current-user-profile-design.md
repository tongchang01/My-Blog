# 后端 V2 当前用户资料设计

## 1. 背景

`identity` 模块已经完成后台登录、access token 校验、refresh token 轮换和全端退出，但 `t_user_info` 尚未落地，当前用户资料接口也尚未开放。

冻结文档已经明确：

- `t_user_auth` 与 `t_user_info` 为 1:1。
- `t_user_info.user_id` 同时作为主键和对 `t_user_auth.id` 的逻辑引用，不建立数据库外键。
- ADMIN 可以管理个人资料；DEMO 只能查看自己的资料。
- 个人简介保留中文、日文和英文三个原始字段。

本设计只完成当前用户资料查询与编辑。修改密码涉及凭据校验、`token_version` 递增和会话失效，必须单独设计和实施。

## 2. 目标

- 落地 `t_user_info` 的领域模型、持久化能力和初始化规则。
- 提供当前登录账号及其完整资料的查询接口。
- 提供 ADMIN 编辑本人资料的部分更新接口。
- 保持账号认证、会话管理和个人资料管理的职责边界清晰。
- 对资料缺失等异常状态保持严格失败，不在运行时静默修复数据。

## 3. 非目标

- 不实现修改密码。
- 不实现头像文件上传，只维护 `avatarUrl`。
- 不实现按用户 ID 查询或编辑资料的管理接口。
- 不实现用户列表、角色管理或访客用户体系。
- 不实现面向博客前台的公开个人资料接口。
- 不在当前用户接口中执行三语简介 fallback。
- 不新增个人资料专用业务错误码。

## 4. 账号与资料初始化

`t_user_auth` 和 `t_user_info` 必须保持严格 1:1：

1. 新建后台账号时，在同一事务中同步创建资料行。
2. 已存在但缺少资料行的历史账号，由迁移脚本一次性补齐。
3. 运行时查询到账号存在而资料缺失时，按数据一致性异常处理。
4. 查询或更新接口不得自动插入资料行，避免掩盖迁移遗漏或数据库损坏。

`nickname` 不可为空。迁移脚本补齐历史资料时，应使用已有可确认的昵称；若没有独立昵称来源，则使用账号用户名作为初始昵称。

## 5. 接口设计

### 5.1 查询当前用户

```http
GET /api/auth/me
Authorization: Bearer <access-token>
```

ADMIN 和 DEMO 均可调用。

响应同时包含账号基础信息与完整资料：

```json
{
  "code": "00000",
  "msg": "success",
  "data": {
    "id": 1,
    "username": "admin",
    "type": "ADMIN",
    "profile": {
      "nickname": "TYB",
      "avatarUrl": null,
      "bioZh": null,
      "bioJa": null,
      "bioEn": null,
      "location": null,
      "website": null,
      "emailPublic": null,
      "githubUrl": null,
      "twitterUrl": null,
      "linkedinUrl": null,
      "zhihuUrl": null,
      "qiitaUrl": null,
      "juejinUrl": null
    }
  }
}
```

不得返回密码摘要、`tokenVersion`、登录失败次数、锁定截止时间或 refresh token 信息。

### 5.2 编辑当前用户资料

```http
PATCH /api/auth/me/profile
Authorization: Bearer <access-token>
Content-Type: application/json
```

仅 ADMIN 可以调用。DEMO 调用时返回 `403 + 10003`，并且不得执行资料查询和数据库更新。

请求示例：

```json
{
  "nickname": "TYB",
  "bioZh": "个人简介",
  "website": "https://example.com",
  "twitterUrl": null
}
```

成功后返回更新后的完整资料对象，不要求前端再次查询。

## 6. PATCH 更新语义

请求字段必须区分三种状态：

| 请求状态 | 更新语义 |
| --- | --- |
| 字段未出现 | 保持数据库原值 |
| 字段出现且有值 | 规范化并更新 |
| 可选字段出现且为 `null` 或空白字符串 | 清空为数据库 `NULL` |

普通 Java `null` 无法同时表达“字段未出现”和“显式传 null”。Web 请求 DTO 必须采用 presence-aware 字段包装或等价的 Jackson 绑定机制，不得通过字符串拼接或不受控的 `Map<String, Object>` 解析请求。

`nickname` 的规则不同：

- 字段未出现时保持原值。
- 字段出现时先去除首尾空白。
- `null`、空字符串或只包含空白均为参数错误，不允许清空。

请求必须至少包含一个受支持字段。空对象或没有产生任何字段意图的请求返回参数校验错误。未知字段一律拒绝，不得静默忽略。

## 7. 字段规则

| 字段 | 规则 |
| --- | --- |
| `nickname` | 必填，去除首尾空白，最大 64 字符 |
| `avatarUrl` | 可空，最大 255 字符，仅接受 HTTP/HTTPS URL |
| `bioZh` / `bioJa` / `bioEn` | 可空，各最大 5000 字符 |
| `location` | 可空，去除首尾空白，最大 64 字符 |
| `website` | 可空，最大 255 字符，仅接受 HTTP/HTTPS URL |
| `emailPublic` | 可空，最大 128 字符，符合邮箱格式 |
| 各社交链接 | 可空，最大 255 字符，仅接受 HTTP/HTTPS URL |

可选字符串规范化后为空时统一写入 `NULL`，不在数据库中保留空字符串。

URL 校验只负责协议和基本格式，不主动访问目标地址，不做站点白名单，也不要求社交链接必须属于对应平台域名。

后台编辑接口返回 `bioZh`、`bioJa` 和 `bioEn` 三个原始值。缺失语言 fallback 到中文属于未来公共展示接口的职责，不在本接口中改写数据。

## 8. 模块结构

在现有 `identity` 模块内增加独立的当前用户资料入口，避免继续扩张会话控制器：

- `CurrentUserController`：处理 `/api/auth/me` 与 `/api/auth/me/profile`。
- 当前用户查询应用服务：组合账号与资料，只负责读取。
- 个人资料更新应用服务：校验角色、合并 PATCH、执行事务更新。
- `UserProfile` 领域模型：表达完整资料及字段规范。
- 资料仓储端口：按 `userId` 查询、加锁查询和更新。
- MyBatis-Plus Entity、XML Mapper 与仓储适配器：实现 `t_user_info` 持久化。

登录、refresh 和 logout 继续留在 `AuthController`，资料服务不依赖 JWT 解析实现，只接收由安全层确认的 `AuthenticatedPrincipal`。

持久化 SQL 必须写在 XML Mapper 中，不得写入 Java 注解。

## 9. 查询数据流

1. `CurrentUserController` 通过 `@CurrentUser` 取得已认证主体。
2. 查询应用服务按主体 ID 查询未删除的 `t_user_auth`。
3. 查询应用服务按同一 ID 查询未删除的 `t_user_info`。
4. 任一记录缺失时抛出内部数据一致性异常。
5. 应用服务组合账号基础信息与资料结果。
6. Controller 转换为稳定的 API VO 并返回。

查询接口不接受客户端传入的用户 ID，防止越权读取其他账号。

## 10. 更新数据流

1. 安全层验证 access token。
2. Controller 取得当前主体并交给更新应用服务。
3. 应用服务首先校验角色必须为 ADMIN。
4. 在事务中按 `userId` 加锁查询未删除的资料行。
5. 将请求中已出现的字段合并到当前完整资料。
6. 对合并后的结果执行领域校验和字符串规范化。
7. 更新 `t_user_info`，由统一审计填充机制维护 `updated_at` 和 `updated_by`。
8. 更新行数必须为 1，否则抛出数据一致性异常并回滚。
9. 返回持久化后的完整资料结果。

加锁查询用于确保并发 PATCH 不会基于同一旧版本互相覆盖不同字段。当前数据规模极小，不引入乐观锁版本列。

## 11. 权限规则

| 操作 | ADMIN | DEMO |
| --- | --- | --- |
| 查询本人账号和资料 | 允许 | 允许 |
| 编辑本人资料 | 允许 | 禁止 |

当前阶段不存在编辑他人资料的接口。角色校验必须在应用层保留，即使 Web 安全配置也对路径设置了角色限制，避免其他调用入口绕过业务边界。

## 12. 错误处理

| 场景 | HTTP | 业务码 |
| --- | --- | --- |
| access token 缺失、无效或失效 | 401 | `10002` |
| DEMO 尝试编辑 | 403 | `10003` |
| 字段非法、空 PATCH 或 JSON 格式错误 | 400 | `90001` |
| 账号或资料缺失、更新行数异常、未预期持久化错误 | 500 | `99999` |

数据一致性异常必须记录中文错误日志，日志包含必要的账号 ID 和失败阶段，但不得记录 access token、密码、完整请求体或其他敏感凭据。对外只返回统一系统错误，不暴露表名、SQL 或内部异常消息。

## 13. 测试策略

### 13.1 领域测试

- 完整资料字段规范化。
- PATCH 字段未出现时保持原值。
- 可选字段显式传 `null` 或空白时清空。
- `nickname` 不允许清空。
- 长度、邮箱和 URL 协议校验。

### 13.2 应用服务测试

- ADMIN 和 DEMO 均能查询本人资料。
- ADMIN 可以部分更新并取得更新后结果。
- DEMO 更新在访问数据库前被拒绝。
- 账号缺失、资料缺失和更新行数异常映射为内部错误。
- 事务异常时不保留部分更新。

### 13.3 Mapper 与仓储测试

- XML SQL 正确读取 `t_user_info`。
- 默认过滤软删除行。
- 加锁查询使用数据库行锁。
- 更新只影响目标 `userId`，并正确填充审计字段。

### 13.4 Web 测试

- `GET /api/auth/me` 响应结构和敏感字段隔离。
- `PATCH` 的未提交、赋值和显式清空三态语义。
- ADMIN、DEMO、未认证和失效 token 的权限边界。
- 参数错误和系统错误的 HTTP 状态及业务码。
- 未知字段遵循全局 JSON 请求约定，不被静默当作有效更新。

### 13.5 MySQL 集成测试

- 新建账号与资料行在同一事务中保持 1:1。
- 历史账号资料补齐脚本可重复验证。
- 并发更新通过行锁串行合并。
- 更新失败时事务回滚。

完成门槛为全量 Maven 测试通过；依赖 Docker 的测试继续遵循项目现有条件执行策略。

## 14. 实施拆分

后续实施计划应拆成小批提交，避免一次提交跨越全部层次：

1. `t_user_info` 领域模型、持久化映射和迁移补齐。
2. 当前用户资料查询。
3. presence-aware PATCH 请求模型与资料更新服务。
4. Web 接口、权限配置和集成测试。
5. API 契约、状态与 roadmap 同步。

每批必须先完成对应测试，再单独使用中文提交信息提交。

## 15. 后续独立任务

个人资料完成后，再单独设计修改密码：

- 校验当前密码。
- 更新 BCrypt 密码摘要。
- 递增 `token_version`。
- 撤销全部 refresh token。
- 明确当前请求是否立即失效及前端重新登录流程。
