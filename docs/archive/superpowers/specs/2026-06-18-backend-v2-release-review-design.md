# 后端 V2 第一版发布前 Review 设计

## 1. 目标

对 `MyBlog-springboot-v2/` 当前实现进行一次发布前只读审查，判断后端第一版是否具备进入前端联调和后续发布准备的工程质量。

本轮不默认设计书正确，也不默认代码正确。每项结论必须同时核对设计要求、实际实现与测试或运行证据，识别以下三类问题：

1. 设计要求本身冲突、过时、不合理或不可执行；
2. 代码偏离有效设计要求；
3. 设计和代码一致，但缺少测试或运行证据证明行为可靠。

## 2. 范围与边界

审查范围：

- `docs/project-handbook/` 中与后端有关的产品、API、架构、规则、ADR、状态和工作流文档；
- `docs/superpowers/` 中仍可能影响当前实现判断的规格、计划和历史 Review；
- `MyBlog-springboot-v2/` 的生产代码、配置、Flyway、Mapper XML、依赖和测试；
- 本地 H2 与 MySQL 验证结果，以及 Docker 缺失导致的 Testcontainers 验证缺口。

边界：

- 不修改后端代码；
- 不修改任何既有设计书、ADR、规则或测试；
- 不执行破坏性数据库操作；
- 只新增最终 Review 报告及本次 Review 的规格、计划文档；
- 修复建议只形成后续批次，不在本轮实施。

## 3. 证据与权威判断

不设置未经核验的单一绝对权威来源。每条要求先确定来源和状态，再判断实现。

### 3.1 文档状态

每条设计要求标记为以下一种：

- `有效`：当前被接受且未被后续决定替代；
- `被替代`：已有更新 ADR、规格或实现决策明确取代；
- `冲突`：同层级文档给出不同要求，无法直接执行；
- `过时`：描述的是已结束阶段或已删除实现；
- `不合理`：要求与当前技术栈、业务目标或维护成本不匹配；
- `证据不足`：无法从现有材料确认其当前效力。

### 3.2 三角核对

每个重要结论至少覆盖三类证据中的两类，Critical 和 Important 必须覆盖三类：

1. 要求证据：产品规则、API 契约、ADR、架构规则或编码规范；
2. 实现证据：Java、配置、SQL、DDL、依赖或资源文件；
3. 验证证据：单元测试、集成测试、ArchUnit、H2/MySQL 运行结果。

文档与代码冲突时，不直接选择任何一方，而是说明冲突、实际风险和建议裁决方向。

## 4. Review 主线

### 4.1 文档治理

- 入口文档、状态、路线图和当前实现是否同步；
- ADR、规则、规格和历史 Review 是否互相矛盾；
- 已被替代的决定是否仍被当作当前要求；
- 文档中的测试数量、模块状态、数据库策略是否过时；
- 注释规范和 SQL 规范是否合理、清晰且可执行。

### 4.2 架构与模块边界

- 模块划分与包结构是否符合当前模块模型；
- domain、application、infrastructure、web 依赖方向；
- 跨模块调用是否只经过 application 边界；
- ArchUnit 是否覆盖全部当前模块和关键禁止依赖；
- common 与通用基础设施是否反向依赖业务模块。

### 4.3 API 与业务契约

- API 路径、请求、响应、分页、排序、错误码与文档是否一致；
- ADMIN、DEMO、匿名用户的读写边界；
- PRIVATE、PASSWORD、DRAFT、SCHEDULED 等内容可见性；
- 多语言标题、补齐规则和稳定排序；
- 评论、附件、友链、站点配置、统计等第一版业务规则。

### 4.4 安全与配置

- JWT、refresh token、token version、密码哈希和撤销语义；
- 公开白名单是否同时约束 HTTP method 和 path；
- DEMO 写操作和敏感读是否由后端强制限制；
- CORS、可信代理、客户端 IP、限流和敏感日志；
- 上传 MIME、大小、路径和访问控制；
- local、test、prod 配置及密钥、数据库凭据默认值。

### 4.5 持久化、SQL 与 Flyway

- Repository 抽象和 Entity/Mapper 隔离；
- BaseMapper、LambdaQueryWrapper、注解 SQL、XML 的使用是否符合场景；
- join、动态条件、聚合、分页、projection、批量 IN 是否进入 XML；
- XML namespace、id、路径、result mapping 和中文业务注释；
- 是否存在 Java 字符串拼 SQL、未受控排序字段或注入风险；
- Flyway 脚本不可变性、版本顺序、MySQL/H2 兼容性；
- 无数据库外键、审计列、软删除、ID、索引、时区和时间精度；
- 事务边界、并发锁、批量操作和 N+1 查询风险。

### 4.6 注释与接口说明

- 先审查现有“必须注释”规则是否过度、过时或与代码结构冲突；
- 对有效规则检查 Entity、DTO、Command、Query、Response、Controller、Application Service、Enum、Mapper 和配置；
- 检查权限、状态流转、审计、限流、时间等特殊逻辑是否说明原因；
- 检查 Javadoc 与 OpenAPI 描述是否一致；
- 区分缺失业务说明与无意义翻译式注释，避免以注释数量代替可维护性。

### 4.7 测试与运行证据

- 核心业务、权限、状态流转、安全、复杂 SQL 的测试覆盖；
- 测试是否只验证 mock，而未验证真实行为；
- H2 与 MySQL 数据类型、SQL、事务和时间精度差异；
- Testcontainers 因 Docker 缺失而跳过的风险；
- 测试隔离、顺序依赖、数据清理和不稳定测试；
- 构建、依赖收敛、架构测试和启动配置验证。

## 5. 严重度与处置标准

### Critical

可能导致未授权访问、凭据泄漏、数据损坏、迁移失败、核心业务错误或无法安全发布。必须在前端联调或发布前优先处理，具体阻塞点在报告中说明。

### Important

会造成接口契约偏差、业务边界错误、MySQL 兼容问题、明显维护风险或关键测试缺口。原则上应在第一版冻结前修复。

### Minor

不影响当前正确性和安全性的可维护性、命名、注释、文档同步或局部测试质量问题，可进入后续整理批次。

不把纯个人偏好、无风险的格式差异或没有证据的猜测列为缺陷。

## 6. Review 执行方式

### 第一遍：建立基线和要求清单

- 记录分支、提交、工作区、Java/Maven/MySQL/Flyway 基线；
- 建立文档状态图和冲突清单；
- 生成模块、API、配置、SQL、Flyway、测试文件清单；
- 将有效要求整理为可追踪检查项。

### 第二遍：自动化扫描

- 使用 `rg`、Maven、ArchUnit 和静态文件统计定位高风险模式；
- 扫描注释语言和关键类型缺失 Javadoc；
- 扫描注解 SQL、XML SQL、Java SQL 字符串和 Mapper 放置；
- 扫描直接时间调用、硬编码密钥、危险配置、跨层和跨模块依赖；
- 扫描 API 鉴权注解、公开端点和测试覆盖映射。

自动化扫描结果仅作为线索，必须人工复核后才能进入报告。

### 第三遍：人工链路审查

按风险优先级追踪关键业务链路：

1. 登录、refresh、logout、改密；
2. ADMIN/DEMO 后台读取与写入；
3. 文章创建、发布、密码访问和公开查询；
4. 评论提交、审核、删除、恢复和通知；
5. 附件上传、引用和删除；
6. 访问统计记录、聚合、清理和 dashboard；
7. Flyway 空库迁移及 H2/MySQL 差异。

### 第四遍：验证和反证

- 对疑似问题查找现有测试；
- 运行无写代码即可完成的聚焦测试或只读验证；
- 对无法验证的项目明确标记限制，不把推测写成事实；
- 去重并检查每项严重度是否与实际影响匹配。

## 7. 最终报告结构

最终新增：`docs/superpowers/reviews/2026-06-18-backend-v2-release-review.md`。

报告包含：

1. 总体结论：是否可进入前端联调，是否可视为发布候选；
2. 审查范围、提交基线、环境和限制；
3. 设计文档有效性与冲突清单；
4. Critical / Important / Minor 发现；
5. 每项发现的设计、代码、测试证据；
6. 已验证且未发现问题的关键边界；
7. Docker/Testcontainers 等未覆盖风险；
8. 按独立提交拆分的修复顺序；
9. 前端联调阻塞项与发布阻塞项。

## 8. 完成标准

- 七条主线均有明确结论；
- 所有 Critical 和 Important 均有三角证据；
- 所有发现均能定位到具体文档或代码文件；
- 文档缺陷和代码缺陷分开记录；
- 明确哪些问题阻塞前端、哪些只阻塞发布；
- 不修改任何被审查对象；
- 工作区除 Review 规格、计划和最终报告外无其它变更。
