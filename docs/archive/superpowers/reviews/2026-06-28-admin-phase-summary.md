# 后台二期阶段总结

> 日期：2026-06-28  
> 分支：`feature/admin-phase-2`  
> 结论：后台已经达到“基本可用”，下一阶段应暂停继续堆后台功能，转向前台联动验收。

## 1. 当前判断

后台现在不是半成品，已经能支撑博客 V2 的核心内容管理。

但它也还不是完整 CMS。真正的问题不再是“后台页面够不够多”，而是：

- 前台是否真的吃到了这些后台数据；
- 后台写出来的数据是否符合前台展示模型；
- 哪些高级配置现在做了也用不上。

所以当前阶段建议冻结后台主线功能，只保留小修和联动验收发现的阻断项。

## 2. 已完成能力

### 2.1 认证与权限

- 后台登录 / refresh / logout。
- ADMIN 可写，DEMO 只读。
- 当前用户资料查询与编辑。
- 本地 MySQL local profile 已可作为开发测试库使用。

### 2.2 仪表盘与统计

- 后台仪表盘接真实统计数据。
- 已有趋势图、热门文章、语言分布等图表。
- 统计数据来自后端 `stats` 模块，不再是纯假数据。

### 2.3 文章管理

- 文章列表。
- 新建文章。
- 编辑文章。
- 删除 / 回收站 / 恢复。
- Markdown 预览。
- 本地草稿。
- 离开确认。
- 文章封面接附件选择器。

### 2.4 分类与标签

- 分类管理。
- 标签管理。
- 新增、编辑、删除、排序等基础能力。

### 2.5 评论管理

- 评论列表。
- 按目标、状态、关键字、删除状态筛选。
- 审核通过。
- 隐藏。
- 删除。
- 恢复。
- 后台回复评论。

说明：后台回复已经存在，但它是否是最终推荐交互还需要和前台评论展示一起验收。

### 2.6 友链管理

- 友链列表。
- 新增、编辑、状态切换、排序、删除。
- 友链头像已接附件选择器。

### 2.7 站点配置

- 站点标题 / 副标题。
- About 多语言内容。
- Logo。
- Favicon。
- ICP。
- Spotify 播放列表 ID。

### 2.8 作者 / 账号资料

- 当前 admin 资料管理。
- 昵称、头像、简介、位置、网站、公开邮箱、社交链接等字段。

注意：这套资料当前更像后台账号资料。前台作者卡片仍主要来自前台静态 author JSON / 默认配置，两者还没有完全统一。

### 2.9 附件管理

- 上传。
- 列表。
- 附件选择器。
- 删除。
- 回收站。
- 恢复。
- 已被文章封面、作者头像、Logo、Favicon、友链头像复用。

## 3. 当前明显缺口

### 3.1 前台作者卡片未完全接后端

前台现在的作者卡片主要来自：

- `frontend/apps/blog/public/api/authors/blog-author.json`
- `frontend/apps/blog/src/features/site-settings/defaults.ts`

后端已有：

- `t_user_info`
- `t_site_config`

但两边模型不是一一对应。

合理方向不是让数据库字段追平 JSON，而是后端提供一个前台 author 聚合接口，拼出前台需要的结构。

### 3.2 前台联动还没有系统验收

需要集中验收这些链路：

- 后台创建文章，前台是否展示。
- 后台改分类 / 标签，前台是否展示。
- 后台改站点配置，前台是否展示。
- 后台改作者资料，前台作者卡片是否展示。
- 后台改友链，前台友链页是否展示。
- 前台提交评论，后台审核 / 回复，前台是否展示。

### 3.3 附件删除缺引用保护

现在附件能删除 / 恢复，但还没有统一判断：

- 文章封面是否引用；
- 作者头像是否引用；
- Logo / Favicon 是否引用；
- 友链头像是否引用。

短期可接受，因为是软删除；但正式使用前应该补“被引用时禁止删除”或至少提示。

### 3.4 前台评论里的站长身份还没最终定

现在后台回复可以写入评论流。

但“站长评论”的最终展示规则还没定：

- 昵称显示 `站长` 还是 `站长：TYB`；
- 是否需要站长徽标；
- 是否使用作者资料头像；
- 是否需要 public comment API 返回 `isAuthor` / `authorRole`。

这件事应和前台评论展示一起做，不建议只在后台继续改。

## 4. 建议后置项

这些现在不建议继续做：

### 4.1 用户 / 角色管理

项目定位是个人博客，目前只有一个 admin。

后台账号只承担管理身份和审计，不需要现在做完整用户管理、角色管理、权限分配 UI。

后置。

### 4.2 主题颜色 / 布局开关 / 导航菜单后台配置

你最终想要全都能管，这是合理目标。

但当前前台还没有完成数据联动验收，过早做主题配置会让后台模型膨胀。

后置。

### 4.3 评论高级互动

包括：

- 后台树形评论管理；
- 访客互相回复的新 UI；
- 表情面板；
- 评论 IP 归属地；
- 站内消息。

这些都不是当前阶段必须能力。

后置。

### 4.4 附件物理清理

当前软删除足够安全。

物理清理要等引用保护和对象存储策略稳定后再做。

后置。

### 4.5 生产部署整理

Docker、反代、生产 env、构建产物发布流程需要做，但不应该和前后台联动验收混在一起。

后置到“功能链路稳定后”。

## 5. 建议保留的小修项

这些可以在联动验收中顺手做：

1. 评论回复作者邮箱策略  
   当前不建议继续围绕 admin 账号展开。更合理方向是后续由前台作者资料统一处理。

2. 附件引用删除提示  
   如果验收时频繁误删图片，就优先补。

3. 后台文档精简  
   已完成的大计划文档可以不再扩展，后续只保留验收记录和缺口清单。

## 6. 下一阶段建议

下一阶段不要继续先做后台功能。

建议切到“前后台联动验收阶段”：

### 6.1 启动三端

- 后端：local profile + `myblog_v2_dev`
- 前台：`frontend/apps/blog`
- 后台：`frontend/apps/admin`

### 6.2 用后台录入真实数据

至少录入：

- 1 篇公开文章；
- 1 篇草稿文章；
- 1 个分类；
- 2 个标签；
- 1 条友链；
- 1 套站点配置；
- 1 套作者资料；
- 1 张文章封面；
- 1 个 Logo / Favicon。

### 6.3 在前台逐页验收

检查：

- 首页；
- 文章详情；
- 分类页；
- 标签页；
- 归档页；
- 友链页；
- 关于页；
- 留言 / 评论区；
- 多语言路径；
- 404 / 空状态。

### 6.4 输出缺口清单

缺口按三类分：

- P0：前台页面打不开或核心数据不显示；
- P1：显示不完整，但不阻断主流程；
- P2：体验优化或可后置配置。

只修 P0 / 必要 P1。

## 7. 阶段结论

后台已经基本可用。

剩下的后台大功能大多可以后置。现在最重要的是回到前台，把真实博客体验跑通。

如果继续孤立开发后台，风险是继续做出“后台能填，但前台不用”的配置。
