`docs/deep-research-report.md` 仍然是后端全量重构的总纲和问题清单，它回答的是“我为什么要重构、旧项目有哪些风险、整体应该覆盖哪些领域”。

当前缺少的不是再写一份新的总纲，而是一份执行期复盘文档：我需要把已经完成的阶段、当前项目结构和持久化策略的依据、尚未处理的风险、下一步决策点统一校准。否则后续继续写代码时，容易出现一种错觉：每个小任务都完成了，但我对整体进度、技术路线和剩余代价失去掌控。

## 当前项目结构说明

当前后端 V2 采用的是模块化单体，而不是微服务。

这是基于当前项目体量和你的明确选择：项目规模不大，三端重构的主要矛盾不是服务拆分，而是先把安全边界、模块边界、数据访问边界、接口契约和代码风格拉齐。如果此时拆成微服务，会提前引入服务发现、远程调用、链路追踪、部署编排、分布式事务和运维成本，收益不匹配。

当前结构按领域拆分，例如：

| 层级 | 作用 |
| --- | --- |
| `modules/identity` | 身份、登录、当前用户、菜单 |
| `modules/content` | 分类、标签、文章、归档、推荐、密码访问 |
| `modules/comment` | 评论读取、提交、后台治理 |
| `infrastructure` | 数据库、配置、安全、跨模块技术适配 |
| `common` | 通用响应、异常、分页等稳定公共能力 |

每个业务模块内部再按 `api`、`application`、`domain`、`infrastructure` 分层。这个结构的依据不是为了形式上贴近 DDD，而是为了达到三个目的：

1. 控制依赖方向，避免 Controller、Service、Mapper、Entity 随意互相调用。
2. 把数据库访问限制在 infrastructure 层，未来替换 ORM 或表结构时不扩散到业务层。
3. 让每个业务域可以按计划迁移，而不是一次性重写所有代码。

## ORM 与持久化策略校准

当前 V2 使用 `JdbcTemplate`，这是迁移阶段的战术选择，不应该被理解成最终技术路线已经永久锁定。

我选择先用 `JdbcTemplate` 的原因是：

1. 初期任务重点是安全基线、模块边界、接口契约和真实数据接入，不是先引入完整 ORM 体系。
2. 旧项目本身表结构和字段命名需要重新审视，过早引入 MyBatis Plus 容易把旧表模型直接固化成新实体。
3. `JdbcTemplate` 透明、轻量、便于在基础迁移阶段明确 SQL 行为，适合先把读侧和关键认证链路跑通。
4. 当前数据库访问被限制在 infrastructure 层，理论上仍保留后续替换为 MyBatis Plus 的空间。

但这里也必须明确：一个项目长期混用多套 ORM 或持久化风格会变奇怪，也会增加维护成本。短期可以因为迁移阶段存在过渡形态，长期必须收敛。

因此后续需要补一个明确决策：后端 V2 最终持久化策略到底是继续使用 `JdbcTemplate`，还是在后台 CRUD 和表结构稳定前切换到 MyBatis Plus。

如果后续采用 MyBatis Plus，我建议遵守以下边界：

| 规则 | 原因 |
| --- | --- |
| `Mapper`、`DO/PO` 只放在 infrastructure 层 | 避免 ORM 模型污染业务层 |
| domain 模型不继承 MyBatis Plus 基类 | 保持领域模型独立 |
| `QueryWrapper` 不进入 api/application/domain | 避免业务逻辑被 ORM 查询 DSL 绑定 |
| 以模块为单位迁移持久化实现 | 避免同一业务链路混杂两套访问方式 |
| 在后台 CRUD 大规模展开前完成选型 | 避免后续批量返工 |

我的建议是：现在先暂停继续扩展评论治理代码，先写一份“持久化策略与 ORM 选型决策”文档，把 MyBatis Plus 是否引入、何时引入、怎么迁移讲清楚。这样比继续推进业务代码更稳。

## 数据库状态

当前真实 MySQL 结构基本没有被 V2 自动改动。

已经发生的主要是：

1. V2 通过本地配置读取你导入的 MySQL 数据。
2. 测试环境使用 H2 + Flyway 构造必要的旧表结构。
3. 部分新增表结构和测试数据存在于测试迁移脚本中，用来保证自动化测试独立运行。
4. 本地 profile 中 Flyway 关闭，避免在没有你确认的情况下改动真实库。

这意味着：现在还没有进入正式的数据库结构重构阶段。后续如果要改现有表结构，应该先做 SQL 级别的结构审计和迁移计划，再决定是手工 SQL、Flyway 版本化迁移，还是先在 V2 独立表中承接。

## 下一步路线建议

我建议下一步不要马上继续写评论审计或新业务接口，而是先做一次持久化策略决策。

原因很直接：你现在的主要担心是项目结构和 ORM 选择已经脱离掌控。如果这个问题不先收敛，后面继续写业务代码只会让你更难判断当前路线是不是正确。

推荐顺序如下：

1. 新增《后端 V2 持久化策略与 ORM 选型决策》文档。
2. 明确是否引入 MyBatis Plus。
3. 明确如果引入，是从新功能开始用，还是先回迁已有 identity/content/comment adapter。
4. 明确真实 MySQL 表结构调整的流程：先审计、再计划、再迁移，不在普通功能任务里顺手改表。
5. 再继续进入评论审计、后台 CRUD 或数据库结构优化。

5. Redis、MQ、自动部署是否暂缓，等核心业务迁移和数据库策略稳定后再引入？

## 当前建议

我的建议是：先不要引入 Redis、MQ 和自动部署，也不要马上改真实表结构。

短期优先级应该是：

1. 先完成持久化策略与 ORM 选型。
2. 再决定是否把已有 `JdbcTemplate` adapter 逐步迁移到 MyBatis Plus。
3. 然后继续评论审计或后台 CRUD。
4. 最后再进入数据库结构优化、上传安全、缓存、MQ、部署自动化等更重的专项。

这样推进的好处是：我既不会推翻已经完成的 V2 基线，也不会在 ORM 和结构路线没讲清楚前继续扩大代码规模。
