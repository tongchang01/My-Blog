# 首个管理员初始化与管理端改密设计

> 状态：已确认，待实现
> 适用范围：MyBlog V2 本地启动、首次生产部署与管理端个人资料页
> 最后校准：2026-07-11

## 目标

让新使用者能在本地直接登录项目，同时让生产环境能够安全地创建首个管理员，并在管理端完成首次改密。

## 已确认约束

- 本地默认账号为 `admin`，默认密码为 `12345678`；它只用于 `local` profile，不是生产凭据。
- 生产环境不在 Flyway SQL、镜像、仓库或公开文档中预置管理员密码。
- 生产首个管理员账号名为 `admin`；密码由临时 root-only `bootstrap.env` 提供。
- 后端已有 `PUT /api/auth/me/password`；本次不新增第二套改密 API。
- 改密页面必须在首个生产管理员登录后即可使用。
- 改密成功后，当前账号的全部 access/refresh token 失效，前端清理会话并返回登录页。
- 不实现完整账号管理、忘记密码、邮箱找回或强制改密字段。

## 方案选择

### 本地：受控默认初始化

`application-local.yml` 启用管理员初始化配置。应用启动时，若不存在有效 ADMIN 账号，则创建 `admin`、BCrypt 密码摘要和空资料；若管理员已经存在，则不覆盖、不重置密码并继续启动。

默认凭据仅用于开发体验。开发文档必须明确：生产不能使用该默认密码。

### 生产：一次性非 Web 初始化

生产正常 API 不启用初始化配置。首次部署时使用同一 API 镜像，以非 Web 模式运行管理员初始化：

1. 临时创建仅 root 可读的 `bootstrap.env`，其中包含管理员账号名和生产初始密码。
2. 通过 Compose 运行一次初始化命令；命令完成后退出。
3. 初始化器只在不存在有效 ADMIN 时创建账号和资料；存在管理员时只报告跳过，绝不覆盖密码或资料。
4. 验证成功后删除 `bootstrap.env`，再启动常规 API 服务。

初始化器不得打印密码、密码摘要或完整环境变量。生产初始密码由部署人员在私有台账保存，并在首个登录后立即通过管理端修改。

### 管理端：复用个人资料页

在现有“个人资料”页面新增一个 ADMIN 专用的密码卡片，不新增菜单或路由。卡片包含：当前密码、新密码、确认新密码。

- 前端先校验非空、新密码长度 8 至 128、两次新密码一致。
- 仅调用现有 `PUT /api/auth/me/password`。
- DEMO 不显示提交入口；后端权限仍是最终约束。
- 成功后不调用 logout 接口，因为改密已使当前 token 失效；前端直接清除本地会话、提示成功并跳转登录页。
- 密码字段只保存在组件内存中；提交结束、页面卸载或失败后不写入日志、URL、store 或 localStorage。

## 数据与接口边界

- `t_user_auth` 和 `t_user_info` 的表结构不增加字段，Flyway 初始化脚本不插入默认账号。
- 新增最小持久化能力：判断有效 ADMIN 是否存在，以及原子创建账号和资料。
- 初始化密码通过已有 `PasswordHashService` 转换为 BCrypt 摘要；不得在 SQL 中保存明文或预计算的生产密码摘要。
- 改密继续使用既有 `ChangePasswordApplicationService`、token version 递增和 refresh token 撤销逻辑。

## 错误处理

- 初始化配置缺少账号或密码、账号名非法、密码长度非法时，初始化命令以非零状态失败且不写入账号。
- 创建时发生并发唯一键冲突或持久化异常时，命令失败；部署人员检查数据库后重试，不删除或重置已有账号。
- 改密接口返回 400、401、403 或 500 时，页面保留可编辑状态并显示现有 API 错误映射；不清理当前会话。
- 仅在接口成功后清理前端会话并重定向登录页。

## 验收与测试

- local 空库启动后可用默认账号登录；重复启动不改变现有密码。
- 生产初始化命令能创建管理员；已有管理员时不覆盖；缺少配置时失败且不写入数据。
- 初始化过程不在日志中输出密码或 BCrypt 摘要。
- 管理端密码卡片对 ADMIN 可见、DEMO 不可提交；请求体与现有 API 契约一致。
- 改密成功后本地会话被清理并跳转登录；当前密码错误、两次新密码不一致和后端拒绝时不清理会话。
- 后端相关单测、MySQL 集成测试、管理端 API/组件测试、typecheck、lint 和 production build 通过。

## 非目标

- 不在本次建立上线后全量前后端能力对齐清单；该工作由 ISSUE-009 在 V2 稳定后单独开展。
- 不修改现有登录、刷新、退出或 token 存储架构。
- 不迁移 V1 账号和密码数据。
