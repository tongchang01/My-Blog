# 文章 Markdown 能力设计

## 目标

把 Markdown 作为文章的正式写作格式：作者提交并保存原始 Markdown，管理端预览与前台文章页遵循同一套语法约定；已有文章中被历史导入丢失的 fenced code block 语言信息可以从原稿恢复。

## 已确认事实

- `C:\Users\TYB\OneDrive\Desktop\blog-series\00-project-structure.md` 中的图表使用了正确的 ` ```mermaid` 围栏。
- 线上 `GET /api/public/articles/2076239691205115905?lang=ja` 返回的 `body` 已经把该围栏变成四空格缩进代码块，因此不含 `mermaid`、`java`、`text` 等语言信息。
- 后端文章表把 `body` 作为原文保存和返回；公开查询没有做 Markdown 转换。
- 前台只使用了基础 `markdown-it`；管理端预览使用手写解析器，两端输出不一致。

## 写作约定

采用 **CommonMark + GitHub Flavored Markdown（GFM）**，并固定以下博客扩展：

| 能力 | 约定 |
| --- | --- |
| 基础 Markdown | 标题、段落、引用、链接、图片、列表、分割线、行内/块级代码 |
| GFM | 表格、删除线、自动链接、任务列表、脚注 |
| 代码块 | 使用 ` ```语言名`；保留语言名并高亮，例如 `java`、`text`、`json`、`sql` |
| 图表 | 使用 ` ```mermaid`，在前台按需加载并渲染 |
| 公式 | 使用 `$...$` 与 `$$...$$`，通过 KaTeX 渲染 |
| HTML | 不允许文章原文注入 HTML，统一转义 |

不把 Obsidian、Pandoc 等私有方言纳入兼容范围；未来如确有具体需求，以单独扩展加入并测试。

## 方案比较

### 方案 A：继续为每种语法增加正则或单独规则

改动很小，但管理端、前台和后续语法会持续分叉，无法解决已有 fenced block 的数据问题。不采用。

### 方案 B：以 markdown-it 为统一基础，按需加载扩展（采用）

保留前台已有的 `markdown-it`，启用其 CommonMark/GFM 能力，增加任务列表、脚注、公式和代码高亮插件；Mermaid 仍仅在文章存在 Mermaid 围栏时动态加载。管理端也使用标准 Markdown 解析器生成预览，不再维护手写解析器。

优点是迁移成本低、支持范围明确、可以在两个应用分别构建；缺点是前台与管理端各自依赖同一套规则，必须用同一份 Markdown 示例契约测试防止漂移。

### 方案 C：替换为完整编辑器或 unified/remark 全家桶

可得到更复杂的编辑体验或 AST 处理能力，但会同时替换稳定的 Markdown 输入方式、增加包体积和迁移风险。当前不需要。

## 运行流程

```mermaid
flowchart LR
    Writer[作者输入原始 Markdown] --> Admin[管理端标准预览]
    Admin --> API[文章 body 原样提交]
    API --> DB[(t_article.body)]
    DB --> PublicAPI[公开文章接口原样返回]
    PublicAPI --> Blog[前台标准渲染]
    Blog --> Mermaid[仅 Mermaid 围栏按需加载]
```

1. 管理端 textarea 保持原始 Markdown，不做格式重写。
2. 管理端预览与前台以同一套语法示例验证输出。
3. 后端继续保存、返回原文，不把 HTML 缓存在文章字段中。
4. 前台先解析为安全 HTML，再在已经挂载的 Mermaid 节点上动态绘制图表。
5. 非法 Mermaid 只保留原代码块，不影响正文其余内容。

## 旧文章恢复

历史内容不是前端可推断的问题：普通缩进代码无法可靠还原其原始语言。迁移必须以 `blog-series` 中的 Markdown 原稿为权威来源。

迁移流程：

1. 根据文章标题和/或 slug 建立“线上文章 ID → 原稿文件”的显式清单。
2. 对每篇文章比较线上 `body` 与原稿，预览差异，不直接覆盖。
3. 只更新已人工确认匹配的文章正文；更新前导出数据库行，更新后通过公开接口复查 ` ```语言` 围栏仍存在。
4. 首批只迁移当前系列文章；其他历史文章保持原样，避免误改。

## 安全与性能

- 原始 HTML 继续关闭，链接保持 `noopener noreferrer` 等安全属性。
- 代码高亮只处理允许的语言名，不执行代码。
- KaTeX 和 Mermaid 都不接受任意脚本；Mermaid 使用严格安全级别。
- Mermaid、KaTeX 和语言高亮资源按文章实际内容拆分加载，普通文章不承担图表资源。

## 验收标准

1. 管理端输入 ` ```java`、` ```text`、` ```mermaid` 后，预览保留语言信息。
2. 保存后，后台详情接口与公开详情接口返回的 `body` 字符串仍包含相同围栏。
3. 前台能正确显示 GFM 表格、任务列表、脚注、公式、代码高亮和 Mermaid 图表。
4. 不含 Mermaid、公式或代码块的文章不会加载相应重量级资源。
5. 含无效 Mermaid 的文章正文仍可正常阅读，错误片段保留为代码块。
6. 原稿迁移后的当前系列文章，公开接口正文与 `blog-series` 原稿逐篇一致。
