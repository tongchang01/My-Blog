# 分支与本地 MySQL 稳定化设计

## 1. 目标

在不移动或删除 V1 目录、不执行正式 V1→V2 数据迁移的前提下，先解决两项会随开发持续放大的问题：

1. 让后端基线、后端接口契约和前端业务提交具备清晰的分支边界。
2. 建立可重复启动的 V2 本地 MySQL 联调环境，同时保留 H2 自动化测试。

完成后继续后台文章详情、编辑、Vditor 和分类/标签管理开发。

## 2. 当前问题

- `frontend-v2-integration` 直接基于 `backend-v2-refactor` 的 `775280e` 创建，因此它天然继承整个 V2 后端，而不是单纯的前端分支。
- 后台文章列表开发期间新增了文章、分类和标签 Web ID 精度契约。这些修改服务于前后端联调，但所有权属于后端接口契约。
- 当前仓库根目录同时保留 V1、V2 后端与前端。此时大规模移动或删除 V1 会产生大量机械 diff，并影响仍可能需要的 V1 维护和后续数据映射。
- H2 适合自动化回归，但不能代替 MySQL 方言、真实 Flyway、字符集、时区和前后端启动联调。
- `docs/project-handbook/migration/` 目前只有迁移骨架，尚未形成表映射、清洗规则和可执行导入脚本。

## 3. 范围

### 本批次包含

- 冻结当前分支引用，确保任何整理步骤都可回退。
- 将后端基线纳入主线集成流程。
- 将前端分支中的后端接口契约提交归入后端所有权。
- 从更新后的主线建立干净的前端替代分支。
- 迁移现有前台、后台、前端相关文档和必要的跨端契约记录。
- 对比替代分支与当前分支的预期文件范围和功能结果。
- 建立本地 V2 MySQL 开发库的配置、初始化和最小种子数据方案。
- 保留并继续运行 H2 测试。

### 本批次不包含

- 删除、移动或重命名 V1 目录。
- 改写 V1 Git 历史。
- 正式导入 V1 线上数据。
- 建立增量同步或长期 ETL。
- 修改已经应用的 Flyway 脚本。
- 合并、推送、创建 PR 或删除旧分支，除非用户在对应动作前明确选择。

## 4. 分支整理设计

### 4.1 安全原则

- 不在现有 `frontend-v2-integration` 上做破坏性 rebase 或 force-push。
- 整理前记录 `master`、`backend-v2-refactor`、`frontend-v2-integration` 的准确提交 SHA。
- 创建明确的备份引用后再开始提交迁移。
- 每次只迁移一个目的明确的提交组，并在组间验证。
- 旧分支保留到替代分支测试、目录差异和提交审计全部通过。

### 4.2 所有权划分

后端所有权：

- `MyBlog-springboot-v2/**`
- 后端 API 契约、schema、认证、迁移与后端测试文档
- 后台文章、分类、标签 ID 的 `string/int64` Web 契约

前端所有权：

- `frontend/apps/blog/**`
- `frontend/apps/admin/**`
- 前台和后台工程内文档
- 只描述前端实现与验收的项目手册更新

跨端文档若同时记录后端接口变化和前端实现结果，应按实际依赖放入接口契约提交或单独文档提交，不为追求目录纯度复制两份。

### 4.3 目标分支形态

```text
master
  ├─ 已集成的 backend-v2 后端基线与接口契约
  └─ clean frontend integration branch
       ├─ 前台工程
       ├─ 后台工程
       └─ 前端规格、计划与验收文档
```

干净前端替代分支从已经包含后端基线与 ID 契约的主线创建。这样 GitHub PR 只显示前端增量，而不会再次展示整套 V2 后端。

### 4.4 提交迁移策略

采用“新分支 + 选择性迁移”，不采用直接重写已推送分支：

1. 先形成后端可独立验证的集成结果。
2. 将后端 ID 契约提交纳入后端集成结果。
3. 从新主线创建前端替代分支。
4. 按前台、后台基础工程、后台文章列表三个阶段迁移前端提交。
5. 对于同时修改前后端的提交，拆出前端部分；后端部分不得在替代前端分支重复出现。
6. 文档按所属阶段迁移，避免把旧状态文档机械照搬。

如提交间依赖过于紧密，允许按最终文件组重新形成等价提交，但必须保留原分支作为审计来源，并在设计记录中列出新旧提交映射。

## 5. 本地 MySQL 设计

### 5.1 测试分层

- H2：单元测试、Web slice、快速集成回归和每次提交的默认验证。
- 本地 MySQL：开发启动、Flyway、Mapper、时区、字符集和前后端真实联调。
- Testcontainers MySQL：Docker 可用时执行的自动化 MySQL 方言守护。

本地 MySQL 不替代 H2；三层各自解决不同风险。

### 5.2 数据库边界

- `myblog_v2_dev`：日常 V2 本地开发与前后端联调。
- `myblog_v2_migration_test`：后续 V1→V2 导入演练，可随时清空重建。
- `myblog_v1_snapshot`：后续从 V1 导出的只读快照，不允许 V2 应用写入。

本批次只建立 `myblog_v2_dev`。另外两个数据库在正式数据迁移规格获批后再创建。

### 5.3 初始化与种子数据

- schema 只由现有 Flyway 管理，不手工复制 DDL。
- 开发种子数据与 Flyway 生产迁移分离，不写入 `V1__init.sql` 或后续生产迁移。
- 最小种子覆盖 ADMIN、DEMO、用户资料、分类、标签、不同状态文章和少量评论。
- 种子数据使用固定业务值以便复现，不复制 V1 隐私数据。
- 本地账号密码、JWT secret、数据库凭据和 stats hash secret 通过环境变量或不入库的本地配置提供。
- 密码种子只允许保存 BCrypt hash，不提交明文生产密码。
- 种子过程应可重复：对空库稳定成功；对非空库明确拒绝或幂等处理，不能静默制造重复数据。

### 5.4 启动和验收

本地 MySQL 基线完成后至少验证：

1. Flyway 在空 `myblog_v2_dev` 上完成迁移。
2. 最小种子导入成功，关键表数量符合预期。
3. ADMIN/DEMO 登录、`/api/auth/me` 和 refresh 正常。
4. 前台公开站点配置、文章列表与详情正常。
5. 后台文章只读列表使用 MySQL 数据，筛选和分页正常。
6. 应用与数据库 session 均使用 Asia/Tokyo。
7. `mvn clean test` 和两个前端工程质量门禁继续通过。

## 6. 错误与回滚

- 后端集成失败：不创建前端替代分支，保留三条现有分支不变。
- 提交迁移出现冲突：停止当前组，记录冲突原因；不通过批量覆盖绕过差异。
- 替代分支测试失败：旧 `frontend-v2-integration` 继续作为可运行基线。
- MySQL 初始化失败：删除并重建 `myblog_v2_dev`，不得修改已冻结 Flyway V1 规避问题。
- 种子导入失败：事务整体回滚；禁止留下半套关联数据。
- 所有清理动作都在替代分支验收和用户选择后执行。

## 7. 后续正式数据迁移

正式 V1→V2 迁移单独设计，前置条件是文章编辑、分类标签、附件与评论链路稳定。届时补齐：

- 表和字段逐项映射
- 丢弃数据清单
- 清洗规则
- 附件文件迁移
- ID 和关联关系策略
- 行数、正文、状态、评论、附件校验报告
- 上线前最终快照和停写窗口

由于 V1 数据量较小，优先采用“全新 V2 库完整重导”，不建立复杂增量同步系统。

## 8. 完成标准

- 主线包含经过验证的 V2 后端基线与后端接口契约。
- 干净前端替代分支不重复携带后端增量。
- 当前前台、后台功能在替代分支上保持一致。
- 旧分支仍可回退，未发生未经确认的删除或 force-push。
- `myblog_v2_dev` 可从空库重复初始化并支撑真实前后端联调。
- H2、MySQL 和前端验证职责清晰并有文档记录。
- V1 目录和 V1 数据保持不变，等待后续独立迁移批次。
