# MyBlog 项目重构执行计划

## 1. 文档目标

本文档用于统一说明 MyBlog 三端工程的当前状态、重构方向、执行优先级、未来功能建议，以及前后台是否值得迁移到 Vue 3 语法糖体系。

本文档偏执行，不是纯讨论稿。重点是明确：

- 先做什么
- 为什么先做
- 哪些事情现在不值得做
- 哪些迁移应当作为独立项目推进

## 2. 项目范围

### 2.1 前台 `MyBlog-blog`

- 技术栈：Vue 3 + TypeScript + Pinia + Vue Router 4 + Element Plus
- 目标：博客展示、文章详情、评论、留言、相册、友链、说说、多语言、主题切换

### 2.2 后台 `MyBlog-admin`

- 技术栈：Vue 2 + Vuex + Vue Router 3 + Element UI
- 目标：文章、分类、标签、评论、用户、权限、说说、相册、友链、网站配置、日志、定时任务管理

### 2.3 后端 `MyBlog-springboot`

- 技术栈：Java 8 + Spring Boot 2.3.7 + Spring Security + MyBatis-Plus
- 目标：提供博客内容、后台管理、鉴权、评论、上传、定时任务、站点配置等业务服务

## 3. 当前状态判断

## 3.1 前台现状

### 优点

- 已经完成 Vue 3 化，基础方向正确
- 页面能力较完整，博客展示效果已有较好基础
- 多语言、主题、音乐播放器、目录导航等能力已经具备

### 问题

- 类型边界薄弱，`any` 较多
- API 响应处理和数据转换重复明显
- 评论相关逻辑通过事件总线串联，耦合较高
- 样式、国际化、主题逻辑彼此影响，修改成本偏高
- 一些页面组件过重，职责不清晰

### 结论

前台不适合“大改版重写”，更适合做渐进式重构。

## 3.2 后台现状

### 优点

- 功能面较全，已经满足日常内容管理需求
- Vue 2 体系成熟，短期维护成本可控
- 业务页面集中，改动面比前台更可控

### 问题

- 技术栈明显偏旧
- Vue 2、Vuex、Element UI 的后续升级空间有限
- 如果继续叠加新功能，技术债会越来越重
- 后续做复杂交互或体验升级时，迁移成本会被继续放大

### 结论

后台值得迁移到 Vue 3，但应当作为独立项目推进，而不是“顺手改语法”。

## 3.3 后端现状

### 优点

- 模块已经覆盖博客系统核心业务
- Redis、RabbitMQ、Quartz、S3、SES 等能力齐全
- 已经具备较强的业务扩展基础

### 问题

- Spring Boot 与部分依赖版本偏旧
- 配置项较多，环境治理复杂
- 模块边界、测试、可观测性仍不够强
- 一些基础设施依赖会提高本地启动和部署门槛

### 结论

后端不建议立刻做大范围业务重写，应优先做基础能力加固和依赖升级规划。

## 4. 总体重构原则

1. 先解决高频维护痛点，再推进技术栈升级
2. 不为“语法统一”单独发起大规模迁移
3. 优先做低风险、高收益的结构收敛
4. 大迁移必须分阶段，不能三端同时大改
5. 文档、类型、测试要跟重构同步推进

## 5. Vue 3 语法糖判断

## 5.1 前台是否值得改成 Vue 3 语法糖

结论：值得，但不应作为独立优先级最高的项目。

原因：

- 前台本身已经是 Vue 3，迁移到 `<script setup>` 的技术阻力不大
- 语法糖有助于改善类型推导、代码可读性和 composable 拆分
- 但前台当前更大的问题是重复逻辑、类型边界和状态耦合，不是“语法长得不够现代”

建议：

- 新组件默认使用 `<script setup lang="ts">`
- 重构到的组件顺手迁移
- 不建议专门开一个“全站只改语法糖”的大工程

## 5.2 后台是否值得改成 Vue 3 语法糖

结论：值得，但前提是先完成 Vue 2 到 Vue 3 的完整迁移方案。

原因：

- 后台当前仍是 Vue 2，不能直接“只换语法糖”
- 真正的迁移会涉及：
  - Vue 2 -> Vue 3
  - Vuex -> Pinia
  - Vue Router 3 -> Vue Router 4
  - Element UI -> Element Plus 或其他组件库
  - 富文本、图表、权限页面等兼容性调整

建议：

- 后台 Vue 3 化应作为单独一期工程
- 不建议现在为了统一风格做半迁移
- 先做后台业务梳理和组件兼容性盘点，再决定实施节奏

## 6. 执行优先级

## 第一阶段：先治理高频维护痛点

目标：降低后续改动成本，不做高风险迁移。

### 前台

- 收敛 API 调用层与响应类型
- 抽离文章列表、评论、分页、Markdown 处理等公共逻辑
- 逐步减少 `any`、`@ts-nocheck`、事件总线耦合
- 修复国际化、主题、样式切换的脆弱点

### 后台

- 盘点页面、模块、第三方依赖、权限体系
- 梳理哪些页面最值得保留，哪些可以重构时合并
- 先做文档化和模块边界整理，不急于整体迁移

### 后端

- 补充模块说明和配置说明
- 明确基础设施依赖和可选依赖
- 梳理安全、日志、异常处理和测试薄弱点

## 第二阶段：准备升级工程

目标：为后台 Vue 3 化和后端升级做技术准备。

### 前台

- 在重构触点中逐步推广 `<script setup>`
- 建立更清晰的 composable 和 domain model

### 后台

- 制定 Vue 3 迁移试点模块
- 验证核心依赖替代方案
- 设计新后台的基础骨架

### 后端

- 制定 Spring Boot 升级路径
- 梳理依赖替换与兼容风险
- 明确测试补齐策略

## 第三阶段：启动结构性升级

目标：推动真正的大升级，但保持节奏可控。

### 前台

- 完成核心公共模块的 Vue 3 风格统一
- 提升可维护性和开发效率

### 后台

- 逐步迁移到 Vue 3 + Pinia + Vue Router 4
- 替换不兼容依赖
- 收敛管理台信息架构和交互体验

### 后端

- 分阶段升级基础框架和关键依赖
- 强化测试、监控和配置治理

## 7. 未来建议新增功能

未来不建议再无序堆功能，应围绕“内容效率、读者体验、运营能力”三条线推进。

## 7.1 内容效率

- 文章草稿协作与版本历史
- 内容定时发布与撤回策略增强
- 更强的 Markdown 组件化能力
- 批量内容导入与迁移工具

## 7.2 读者体验

- 更稳定的多语言内容展示策略
- 阅读偏好设置
- 搜索体验优化
- 文章系列、专题推荐、导航聚合页
- 更完整的移动端阅读体验优化

## 7.3 运营能力

- 访问分析面板增强
- 评论风控和审核策略增强
- 站点健康检查与告警
- 更细粒度的权限与操作审计

## 8. 不建议现在做的事情

- 不建议三端同时发起大规模重构
- 不建议前台单独为了语法糖做全量迁移
- 不建议后台在 Vue 2 架构上继续叠大量长期功能
- 不建议后端在没有测试和回归策略时直接激进升级主框架

## 9. 建议落地顺序

### 近期

1. 前台完成高频维护点治理
2. 完善中文 README 与执行计划文档
3. 盘点后台迁移清单
4. 梳理后端配置、依赖和升级风险

### 中期

1. 启动后台 Vue 3 迁移方案设计
2. 推进前台 composable、类型和组件收敛
3. 启动后端升级准备和测试补齐

### 长期

1. 后台完成 Vue 3 化
2. 后端完成主要依赖升级
3. 逐步上线更强的内容、运营和读者体验能力

## 10. 最终建议

当前最合理的路线不是“马上全面重构”，而是：

- 前台先做渐进式治理
- 后台把 Vue 3 迁移作为独立升级工程
- 后端先补基础能力，再做框架升级

这条路线的优点是风险更低、收益更稳定，也更符合当前项目已经在线运行、功能完整但技术债逐渐累积的状态。
