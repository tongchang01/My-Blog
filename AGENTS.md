# AI 协作规则

> 本文件是 AI 编码助手在仓库根目录下的工作约束。当前文档入口见 `docs/README.md`，开发手册见 `docs/handbook/`。

## 任务与提交拆分

- 新建任务分支时不使用 `codex/` 前缀，按 `docs/governance/branch-policy.md` 使用普通分支命名，例如 `feature/<scope>-<short-name>`、`fix/<scope>-<short-name>`、`docs/<scope>-<short-name>`。
- 项目开发文档、计划文档、review 文档和提交说明默认使用中文；除非用户明确要求英文，不得用英文撰写正文。
- 开始实现前，必须把任务拆成可以独立实现、独立验证、独立回滚的小任务。
- 每个 Git 提交只完成一个明确目的，不得把清理、基础设施、业务模块重建、文档整理等不同阶段混在同一次提交。
- 优先按一个组件、一个基础能力或一个紧密相关的文件组提交；完成一个小任务并验证后再进入下一项。
- 提交前必须检查 `git diff --stat` 和 `git status --short`。如果变更扩散到大量文件，先继续拆分任务和提交；批量删除、机械迁移等确实不可再拆的情况，必须提前向用户说明范围和原因。
- 不得为了减少提交次数而积累数百个文件后一次性提交。
- Git 提交信息必须使用中文，并准确描述本次提交的单一目的。
- 未经用户明确要求，不自动合并、压缩或改写已经完成的小提交。

## 验证

- 每个小任务在提交前运行与其风险范围匹配的测试。
- 阶段结束时再运行 `mvn clean test` 等完整验证，不能用阶段级验证代替每个小任务的局部验证。

## 实现选型

- 能用标准库或平台原生能力解决的问题，不得重复手写实现。
- 标准库不足时，优先使用框架、语言或既有依赖的官方文档推荐方案；不得为了可控感重复造轮子。
- 只有现有标准能力和官方方案都不能满足需求时，才允许自定义实现，并在提交说明中说明缺口和取舍。
